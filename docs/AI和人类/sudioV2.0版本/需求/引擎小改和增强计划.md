# 22. ScheduleCard 与 EffectSink 确认语义

本文用于补齐本轮引擎审查后确认的两个语义点：

1. `schedule_recurring_call` 不能生成无 CallCard 的周期电话。
2. `EffectSink` 不能 fire-and-forget，必须确认成功后才能把 effect 标记完成。
3. 一个 StoryPackage 视为一个章节；章节切换由结束点配置驱动，并清理旧剧情卡。

相关文档：[需求/00](../需求/00-文档索引.md) · [需求/01](../需求/01-CallCard数据模型.md) · [需求/02](../需求/02-运行时协议.md) · [需求/16](../需求/16-错误处理与幂等事务.md) · [07](./07-壳嵌入与导出契约.md) · [19](./19-引擎宿主与会话模型.md)

## 1. 背景

本仓铁律是：**每一通电话必须由一张 CallCard 驱动；无剧情卡时才 fallback 到 FreeCallCard。**

当前已完成的主链可以支撑剧情通话、自由通话、一次性延迟外呼、WET、Memory、EffectExecutor。但审查发现两个需要定稿的边界：

- 周期性外呼任务如果没有 `cardId/packageId`，会变成“只有 topicHint 的裸任务”，后续无法进入某张 CallCard。
- 媒介 effect 如果只通知壳而不等待壳确认，`EffectPlanResult` 会把尚未完成甚至可能失败的副作用记为成功。
- 当前 `end_story` 只标记当前包完成，尚未承载“清理旧章节状态并安排下一章节入口”的章节切换语义。

这两个点都会削弱“CallCard 驱动”和“挂机后 EffectExecutor 可靠推进”的核心约束，因此需要明确修正。

## 2. 结论

### 2.1 剧情里只有一次性延迟外呼

剧情是线性/分支流程，不是循环任务系统。

剧情通话中允许：

```text
A 角色本通结束
  → effect: schedule_call_card
  → 等待 minMs / delayMinutes
  → B 角色主动外呼用户
  → 进入明确的剧情 CallCard
```

例如：

```text
澜星姐姐：我等下去和张老板说一下内存条的事，让他等下打给你。
挂机后：
  attach/schedule 张老板名下的某张剧情卡
  过几分钟张老板外呼用户
```

该 effect 必须包含：

- `agentId`
- `packageId`
- `cardId`
- `minMs` 或 `delayMinutes`

禁止只靠 `topicHint` 推进剧情。

### 2.2 周期性外呼归 ScheduleCard

周期性外呼不是 StoryPackage 内的剧情循环，而是角色日常/系统调度能力。

引入概念：**ScheduleCard**。

ScheduleCard 是 FreeCard 的超集：

- 它仍然是一张 CallCard。
- 它属于某个角色。
- 它继承 FreeCard 的自由通话能力和工具能力。
- 它额外带调度元信息：周期、时间窗、冷却、优先级、是否受 ActiveStoryLock 压制等。
- 触发后仍然按 CallCard 通话流程进入 `beginCall/endCall`。

因此，周期性任务也不能是裸任务。它必须能解析到一张 CallCard：

```text
recurring schedule
  → scheduleCardId
  → resolve ScheduleCard
  → agent_outbound
  → CallSession.frozenCard
```

### 2.3 EffectSink 必须确认完成

EffectSink 是壳侧媒介副作用端口，不要求引擎自己播 WAV 或操作硬件。但引擎必须知道壳侧副作用是否成功接收/生成完成。

正确语义：

```text
EffectExecutor
  → 执行逻辑 effect
  → 如果 effect 需要壳侧媒介能力，await EffectSink
  → Sink 成功：effect = executed
  → Sink 失败：effect = failed
  → critical=true：中止后续 effect
  → 返回 EffectPlanResult
```

错误语义：

```text
EffectExecutor
  → 通知壳
  → 不等待结果
  → 直接标记 executed
```

后者会导致剧情状态与壳侧状态不一致。例如语音信箱没创建成功、系统提示音没生成成功，但剧情已经认为完成。

### 2.4 StoryPackage 等价章节；只做结束点，不做显式开始点

本仓不需要单独设计 `startStory(packageId)` 作为章节启动 API。章节的启动由上一章节的结束点安排。

定稿语义：

```text
一个 StoryPackage = 一个章节
当前章节完成
  → 命中终局 exit
  → end_story
  → ChapterTransitionExecutor
      1. 标记当前章节 completed
      2. 释放 ActiveStoryLock
      3. 清理所有角色的旧故事 pending cards
      4. 用户与任意角色通话回到 FreeCallCard
      5. 若结束点配置了下一章节，则按配置挂下一章节第一张卡
```

也就是说：**开始点不是独立机制，结束点才是章节切换的唯一入口。**

章节切换期间，所有旧章节的故事卡必须清掉。否则用户随机拨打任意角色时，可能被旧 pending 污染，误入已经结束的剧情状态。

## 3. ScheduleCard 设计

### 3.1 FreeCard / StoryCard / ScheduleCard 分工

| 类型 | 用途 | 触发方式 | 是否剧情线性流程 |
|------|------|----------|------------------|
| FreeCard | 无剧情卡时的自由通话 fallback | 用户拨号 / 被动 fallback | 否 |
| StoryCard | 剧情中的明确一通电话 | 用户呼入 / 角色外呼 / 调试 simulate | 是 |
| ScheduleCard | 角色日常/周期性主动外呼 | schedule tick | 否 |

ScheduleCard 不替代 StoryCard。剧情里的“过几分钟让某人打给你”仍使用 `schedule_call_card` 指向 StoryCard。

### 3.2 ScheduleCard 是 FreeCard 超集

建议数据形态：

```ts
type CardKind = "free" | "story" | "system" | "schedule";

interface ScheduleCardDefinition extends CallCardDefinition {
  cardKind: "schedule";
  ownerAgentId: string;
  entryMode: "outbound_auto";
  interactionMode: "realtime_dialogue" | "playback_only" | "hybrid";
  schedule: {
    mode: "daily" | "weekly";
    hour?: number;
    minute?: number;
    weekdays?: number[];
    timeWindow?: {
      startHour: number;
      endHour: number;
    };
    cooldownMs?: number;
    priority?: number;
    blockedByActiveStoryLock?: boolean;
  };
}
```

最小 v1 可以不一次性实现所有字段，但必须保证：

- `cardKind: "schedule"` 或等价结构存在。
- 周期任务触发时能解析到 `scheduleCardId`。
- ScheduleCard 触发后走正常 CallCard 流程，而不是 topicHint 裸跑。

### 3.3 recurring intent 形态

当前 `schedule_recurring_call` 不应继续允许缺少卡引用。

建议改为二选一：

```ts
type RecurringScheduledIntent =
  | {
      kind: "recurring";
      intentId: string;
      agentId: string;
      scheduleCardId: string;
      hour: number;
      minute: number;
      scheduleMode: "daily" | "weekly";
      weekdays?: number[];
      status: "active" | "paused" | "cancelled";
    }
  | {
      kind: "recurring";
      intentId: string;
      agentId: string;
      packageId: "__free__";
      cardId: string; // 角色 freeCardId 的显式快照，或由 resolver 实时解析
      hour: number;
      minute: number;
      scheduleMode: "daily" | "weekly";
      weekdays?: number[];
      status: "active" | "paused" | "cancelled";
    };
```

更推荐第一种：`scheduleCardId`。这样不会把 FreeCard 与 ScheduleCard 混在同一个概念里。

### 3.4 一次性延迟外呼卡允许用户提前呼入

一次性延迟外呼不是“只能外呼”的卡，而是“目标角色已经知道这件事，准备稍后主动联系用户”的剧情状态。

典型场景：

```text
1. 澜星姐姐：我等下去和张老板说一下内存条的事，让他等下打给你。
2. 用户挂断。
3. 系统把张老板的后续剧情卡挂到张老板名下，并开始计时。
4. 用户不想等，直接打给张老板。
5. 张老板接起：我正准备给你打呢，澜星刚跟我说了内存条的事。
```

因此，`schedule_call_card` 产生的这张后续卡应是**双入口剧情卡**：

| 实际触发 | 角色姿态 |
|----------|----------|
| 定时到期，角色外呼用户 | “我听澜星说了内存条的事，给你回个电话。” |
| 用户提前打给角色 | “我正准备给你打呢，澜星刚跟我说了内存条的事。” |

这要求引擎区分两个概念：

- **推荐触发方式**：这张卡原本计划由角色外呼。
- **允许触发方式**：用户也可以提前呼入消费同一张卡。

建议形态：

```ts
entry: {
  preferred: "outbound_auto";
  allowed: ["outbound_auto", "inbound_user_dial"];
}
```

若暂时不改大 schema，可用兼容形态：

```ts
entryMode: "either";
activationHint: "outbound_auto";
```

但无论采用哪种形态，运行时都必须记录本通真实入口：

```ts
actualEntry: "inbound_user_dial" | "outbound_auto";
```

`Composer` 应根据 `actualEntry` 决定 `ComposeScene.callDirection` 与开场层，而不是只根据卡定义的 `entryMode` 推断。

#### 3.4.1 schedule_call_card 的运行时语义

`schedule_call_card` 执行后，应同时完成两件事：

```text
EffectExecutor
  → 在目标角色 board 下挂同一张 pending CallCard
  → 创建 schedule once intent
  → schedule intent 引用该 pending instanceId
```

建议 pending 结构补充：

```ts
interface CallCardInstance {
  instanceId: string;
  cardId: string;
  packageId: string;
  agentId: string;
  status: "pending" | "active" | "completed" | "cancelled";
  entryMode?: "either" | "inbound_user_dial" | "outbound_auto";
  activationHint?: "outbound_auto" | "inbound_user_dial";
  scheduledIntentId?: string;
  createdAt: string;
  updatedAt?: string;
}
```

建议 once intent 结构补充：

```ts
interface ScheduledOnceIntent {
  kind: "once";
  intentId: string;
  agentId: string;
  cardId: string;
  packageId: string;
  linkedInstanceId: string;
  fireAtMs: number;
  status: "pending" | "fired" | "consumed" | "cancelled";
}
```

#### 3.4.2 用户提前呼入时的消费规则

当用户在定时器触发前主动拨打目标角色：

```text
user_dial(agentId)
  → pending selector 可命中 entryMode=either 的延迟外呼卡
  → beginCall 时 actualEntry = inbound_user_dial
  → linked schedule intent 标记 consumed/cancelled
  → 后续 tick 不再触发重复外呼
```

当定时器正常触发：

```text
advanceClock / tick
  → 找到 once intent
  → 如果 linked pending 已 completed/cancelled/consumed：跳过
  → 否则 agent_outbound 命中同一张 pending 卡
  → beginCall 时 actualEntry = outbound_auto
```

这可以避免同一张剧情卡被消费后，几分钟后又重复外呼。

#### 3.4.3 卡内容编写约束

延迟外呼目标卡的上下文应同时支持两种开场姿态：

```text
privateBrief:
  澜星已经把事情告诉张老板。若用户提前打来，张老板要表现为“正准备打给你”。

promptScenes:
  - match actualEntry=inbound_user_dial
    openingSpeakable="我正准备给你打呢，澜星刚跟我说了内存条的事。"
  - match actualEntry=outbound_auto
    openingSpeakable="我听澜星说了内存条的事，给你回个电话。"
```

如果暂时没有 `actualEntry` matcher，也应至少把这两种姿态写入 private/speakable brief，让模型知道两种入口都合法。

### 3.5 剧情 effect 约束

剧情卡出口里：

- 允许 `schedule_call_card`，用于一次性延迟外呼。
- 不建议允许 `schedule_recurring_call`。
- 如果仍保留 `schedule_recurring_call`，它必须只创建 ScheduleCard 任务，不能指向剧情 StoryCard 循环。

建议校验规则：

| ruleId | level | 触发 |
|--------|-------|------|
| `SCHEDULE_ONCE_CARD_REQUIRED` | error | `schedule_call_card` 缺 `agentId/packageId/cardId` |
| `SCHEDULE_ONCE_EARLY_INBOUND` | warning/error | 延迟外呼目标卡不能被用户提前呼入消费 |
| `SCHEDULE_LINKED_INSTANCE` | error | once intent 未关联目标 pending instance，无法避免重复外呼 |
| `SCHEDULE_RECURRING_CARD_REQUIRED` | error | `schedule_recurring_call` 缺 `scheduleCardId` 或明确 free/schedule 卡引用 |
| `SCHEDULE_RECURRING_IN_STORY` | warning/error | StoryCard 内出现 recurring effect |
| `SCHEDULE_CARD_KIND` | error | recurring 指向的卡不是 `schedule` 或允许的 free fallback |

## 4. 章节结束点与下一章节安排

### 4.1 一个 StoryPackage 就是一个章节

当前定稿：

- `StoryPackage` 不再只是“内容包”，运行语义上也等价一个章节。
- 章节内部仍由 CallCard + Exit + Effect 推进。
- 章节之间不靠卡链自然散落推进，而靠章节结束点统一切换。
- 不做显式 `startStory`；下一章节的启动由上一章节的 `end_story` 配置决定。

### 4.2 end_story 承载章节切换配置

推荐把章节切换配置放在 `end_story` effect 上：

```json
{
  "id": "end_chapter_01",
  "effect": "end_story",
  "reason": "chapter_completed",
  "cleanup": {
    "clearStoryCards": "all",
    "preserveFreeCards": true
  },
  "next": {
    "packageId": "chapter_02",
    "agentId": "zhang_boss",
    "cardId": "zhang_boss_entry",
    "activation": "delay",
    "delayMs": 300000,
    "entryMode": "either",
    "activationHint": "outbound_auto"
  }
}
```

字段语义：

| 字段 | 含义 |
|------|------|
| `cleanup.clearStoryCards` | 章节结束后清理哪些 story pending cards；v1 固定 `"all"` |
| `cleanup.preserveFreeCards` | FreeCard 是角色基础能力，不应被清理；v1 固定 `true` |
| `next.packageId` | 下一章节 StoryPackage id |
| `next.agentId` | 下一章节第一张卡挂到哪个角色名下 |
| `next.cardId` | 下一章节第一张 CallCard |
| `next.activation` | 下一章节入口触发方式 |
| `next.delayMs` | `activation="delay"` 时的延迟 |
| `next.entryMode` | 下一章节入口 pending 的允许入口 |
| `next.activationHint` | 推荐入口姿态，如计划外呼 |

### 4.3 activation 类型

```ts
type ChapterNextActivation =
  | "immediate"
  | "delay"
  | "wait_user_dial";
```

| activation | 行为 |
|------------|------|
| `immediate` | 立刻把下一章节入口卡挂到目标角色；如需要外呼，可由壳/调试器立即触发 `agent_outbound` |
| `delay` | 创建 once schedule；同时按 3.4 规则挂可提前呼入的 pending |
| `wait_user_dial` | 只挂 pending，等待用户拨打目标角色进入下一章节 |

如果 `end_story.next` 不存在，则章节结束后世界回到纯 Free 状态：所有角色无 story pending，用户拨任意角色都走 FreeCallCard。

### 4.4 清场铁律

章节结束后必须清理旧剧情卡：

```text
for every agent in profile.callCards.board.byAgent:
  remove pending item where packageId != "__free__"
```

注意：

- FreeCard 不在 board pending 里，仍由 CharacterDef.freeCardId 提供 fallback。
- 旧章节的 schedule once intent 也应被取消或标记 `cancelled`，避免结束后又触发旧章节外呼。
- 如果清场后要安排下一章节入口，应先清旧卡，再挂新卡。
- 清场不应删除长期 Memory、world facts、NPC knowledge；这些是用户经历的一部分。

### 4.5 章节状态

当前章节：

```text
profile.stories[currentPackageId].status = "completed"
profile.stories[currentPackageId].endedAt = nowIso
releaseStoryLock(currentPackageId)
```

下一章节若被安排：

```text
profile.stories[nextPackageId].status = "active" 或 "inactive_pending"
profile.stories[nextPackageId].plannedEntry = {
  agentId,
  cardId,
  activation,
  delayMs?
}
```

状态命名可以由实现微调，但必须能表达：

- 当前章节已完成。
- 下一章节入口已被安排。
- 用户在等待期间拨任意非入口角色，应走 FreeCallCard。

### 4.6 与 ActiveStoryLock 的关系

- `end_story` 必须释放当前章节的 ActiveStoryLock。
- 如果下一章节入口被安排，是否立刻建立新 lock 由下一章节配置决定。
- 默认建议：下一章节真正 `beginCall` 时再激活 StorySave / lock，避免章节间等待期锁住用户日常自由通话。

## 5. EffectSink 确认语义

### 5.1 当前需要纠正的点

EffectSink 不负责剧情逻辑，但负责壳侧媒介副作用，例如：

- `create_voicemail`
- `play_system_prompt`
- 后续可能的 LED、提示音、外设状态变化

这些副作用不应该“发出去就算成功”。

### 5.2 正确接口

建议接口保持 Promise 语义：

```ts
interface EffectSink {
  applyMediaEffect(input: EffectSinkApplyInput): Promise<EffectSinkResult>;
}

type EffectSinkResult =
  | {
      ok: true;
      sinkEventId?: string;
      payload?: Record<string, unknown>;
    }
  | {
      ok: false;
      error: string;
      code?: string;
      retryable?: boolean;
    };
```

也可以允许同步实现，但 Executor 内部统一 `await Promise.resolve(...)`。

### 5.3 EffectExecutor 时序

正确时序：

```text
for effect in selectedExit.effects:
  if previous critical failed:
    mark skipped
    continue

  apply logical profile/session mutation

  if media effect:
    result = await effectSink.applyMediaEffect(...)
    if result failed:
      mark failed
      rollback? no automatic rollback in v1
      if critical: abort remaining
      continue

  write effectLedger executed
  mark executed
```

注意：

- v1 不做自动回滚。
- 非 critical 失败默认继续。
- critical 失败中止后续 effect。
- 只有 Sink 成功后才写 `effectLedger.executed`。
- 壳侧 LED/状态变化应基于最终 `EffectPlanResult`，不要乐观更新。

### 5.4 executeEffects 改为 async

需要连锁调整：

```text
executeEffects(...) → Promise<EffectPlanResult>
runFreeCallPostPipeline(...) await executeEffects
EngineHost.endCall(...) await executeEffects
相关测试改 await
```

## 6. Cursor 修改清单

### 6.1 ScheduleCard / recurring

建议修改：

1. 在 CallCard schema 中增加 `cardKind: "schedule"`，或新增 ScheduleCard schema。
2. 在角色定义或 schedule 目录中明确 ScheduleCard 存放位置。
3. `schedule_recurring_call` 不允许缺卡引用。
4. 剧情 StoryCard 中 recurring effect 至少 warning，最好 error。
5. `tickScheduleOnce/materializeRecurringOccurrences` 物化 recurring 时，必须产出可 resolve 的 `agent_outbound` pending。
6. 无 `cardId/packageId/scheduleCardId` 的 recurring intent 不再静默保留为可观测行；应校验失败或迁移为 disabled。
7. `schedule_call_card` 保持一次性延迟外呼，必须 `agentId/packageId/cardId`。
8. `schedule_call_card` 应在目标角色 board 下立即挂 pending，并让 schedule once intent 关联该 pending instance。
9. 延迟外呼 pending 应支持用户提前呼入消费；推荐 `entryMode=either + activationHint=outbound_auto`，或新 schema 的 `entry.preferred/allowed`。
10. `beginCall` 应记录本通真实入口 `actualEntry`，Composer 用它决定 callDirection / opening。
11. 用户提前呼入并开始/完成该 pending 后，关联 once intent 必须标记 `consumed` 或 `cancelled`，后续 tick 不再重复外呼。

### 6.2 章节结束点

建议修改：

1. `end_story` 支持 `cleanup` 与 `next` 配置。
2. `end_story` 执行时标记当前 `StoryPackage` completed，并释放 ActiveStoryLock。
3. `end_story` 执行时清理所有角色 board 下旧 story pending cards。
4. 清理旧章节 schedule once intent，避免结束后旧章节继续外呼。
5. 若存在 `next`，按 `activation` 安排下一章节第一张卡：
   - `immediate`：挂 pending，允许壳/调试器立即外呼。
   - `delay`：挂 linked pending + once schedule。
   - `wait_user_dial`：只挂 pending 等用户拨打。
6. 下一章节等待期间，非入口角色没有 story pending 时必须走 FreeCallCard。
7. 下一章节真正 `beginCall` 时再激活 StorySave / lock，除非配置明确要求提前锁。

### 6.3 EffectSink

建议修改：

1. `EffectSink.applyMediaEffect` 返回可等待结果。
2. `executeEffects` 改 async。
3. 对 media effect 必须 await sink。
4. sink 失败时 effect result = `failed`。
5. sink 失败时不得写 `effectLedger.executed`。
6. `critical: true` 的 media effect 失败后，中止后续 effect。
7. `EffectPlanResult.status` 正确返回：
   - 全成功：`completed`
   - 非 critical 失败：`completed_with_errors`
   - critical 失败：`aborted`
8. 日志/WET 中记录 `planStatus` 与失败项。

## 7. 验收测试建议

### 7.1 recurring 不允许裸任务

测试点：

- `schedule_recurring_call` 缺 `scheduleCardId/cardId` -> validatePackage error。
- recurring tick 不会生成无 CallCard 的 outbound pending。
- ScheduleCard 触发后能 `resolve(agent_outbound)` 并进入 `beginCall`。

### 7.2 剧情延迟外呼

测试点：

```text
StoryCard A endCall
  → selectedExit effects: schedule_call_card(B.card)
  → advanceClockToNextIntent
  → fired includes B.card
  → resolve(agent_outbound: B)
  → beginCall frozenCard.cardId === B.cardId
```

### 7.3 用户提前呼入消费延迟外呼卡

测试点：

```text
StoryCard A endCall
  → schedule_call_card(B.card, delay=5min)
  → profile.callCards.board.byAgent[B].pending includes B.card
  → pending.entryMode supports inbound_user_dial + outbound_auto
  → user_dial(B)
  → resolve hits B.card, not FreeCard
  → beginCall actualEntry === inbound_user_dial
  → linked once intent status = consumed/cancelled
  → advanceClock(5min)
  → no duplicate outbound pending / no duplicate fired call
```

另测正常外呼路径：

```text
StoryCard A endCall
  → schedule_call_card(B.card, delay=5min)
  → advanceClock(5min)
  → agent_outbound(B)
  → resolve hits same linked pending B.card
  → beginCall actualEntry === outbound_auto
```

### 7.4 章节结束点清场并安排下一章节

测试点：

```text
Chapter01 最后一张卡 endCall
  → selectedExit effects: end_story(next=chapter02.entry)
  → profile.stories.chapter01.status === completed
  → ActiveStoryLock chapter01 released
  → all old story pending cards cleared
  → old schedule once intents cancelled
  → chapter02 entry card 按配置挂到目标角色
```

如果没有 `next`：

```text
end_story(no next)
  → all story pending cleared
  → user_dial(any agent)
  → resolve source === free
```

如果 `next.activation="delay"`：

```text
end_story(next delay)
  → linked pending + once schedule created
  → user can call entry agent early
  → early call consumes linked schedule
```

### 7.5 EffectSink 必须等待成功

测试点：

- Recording sink 成功：media effect = `executed`，ledger 写入。
- Failing sink 非 critical：media effect = `failed`，plan = `completed_with_errors`，后续 effect 继续。
- Failing sink critical：media effect = `failed`，后续 effect = `skipped`，plan = `aborted`。
- sink 返回 Promise 延迟 resolve：`endCall` 必须等待后才返回。
- sink 返回 Promise reject：effect = `failed`。

## 8. 非目标

- 不要求引擎真实播放 WAV。
- 不要求引擎直接控制 LED/GPIO。
- 不要求剧情系统支持循环剧情。
- 不做单独 `startStory` API 作为章节入口；章节启动由上一章节结束点安排。
- 不做 effect 自动回滚。
- 不把 ScheduleCard 做成 StoryPackage 内的循环分支节点。

## 9. 最终口径

引擎仍保持纯逻辑核心：

```text
CallCard 决定一通电话是什么
Exit 决定挂机后走哪里
EffectExecutor 决定状态如何推进
EffectSink 让壳侧副作用可确认
ScheduleCard 负责非剧情的周期性外呼
end_story 负责章节收束与下一章节安排
```

因此：

- 剧情里只有明确卡驱动的一次性延迟外呼。
- 周期性外呼必须由 ScheduleCard 或明确 Free/Schedule 卡驱动。
- 一个 StoryPackage 等价一个章节；章节切换由结束点配置驱动。
- 章节结束后必须清理旧 story pending，世界回到 Free 或进入下一章节入口等待态。
- 媒介副作用不由引擎亲自执行，但必须由壳确认成功后，引擎才能把 effect 标记完成。
