# 故事包创建、导入、导出界面设计导向稿

> 版本定位：本文定义故事包新建、导入、导出、版本检查相关界面的设计方向。  
> 这些流程应当是轻量任务流，不应变成完整后台表单系统。

## 1. 核心原则

故事包是 Studio V2.0 的主要工程单位，也可以理解为一个章节。

创建、导入、导出必须遵守：

1. 用户填写人类可读信息，系统生成内部 ID。
2. 创建流程短，不把用户卡在配置细节里。
3. 导入流程先检查，再确认，不直接污染当前工程。
4. 导出流程先校验，再打包，不导出明显不完整内容。
5. 版本、schema、兼容性问题要用人话表达。

## 2. 新建故事包

新建故事包应使用浮层或独立轻量页面。

第一版字段：

- 故事包名称。
- 简短描述。
- 默认语言。
- 初始角色选择，可为空。
- 是否创建默认起点卡。

系统自动生成：

- packageId。
- 起点卡 ID。
- 章节结束节点 ID。
- 默认 FreeCard ID，如需要。

不要求用户填写：

- packageId。
- schemaVersion。
- createdAt。
- updatedAt。
- runtime 状态。

## 3. 新建后的默认结构

新建故事包完成后，应直接进入故事编辑器。

默认画布建议包含：

- 一个章节开始节点。
- 一个章节结束节点。
- 如果选择了初始角色，可以创建第一张空 CallCard。
- 左侧角色锚点栏显示已选择角色。

第一张卡不是必须立刻完整配置，但应该有明显的待完善状态。

## 4. 导入故事包

导入流程建议分三步：

```text
选择文件 -> 预检报告 -> 确认导入
```

### 4.1 选择文件

支持：

- 本地 JSON。
- 故事包压缩包。
- 后续可扩展远程导入。

界面要求：

- 支持拖拽上传。
- 显示文件名和大小。
- 不在选择后立刻写入工程。

### 4.2 预检报告

预检报告需要展示：

- 故事包名称。
- schema 版本。
- 卡片数量。
- 角色数量。
- 资源数量。
- 是否缺资源。
- 是否存在未知 effect。
- 是否存在不可达卡片。
- 是否存在 ID 冲突。
- 是否需要迁移。

预检结果分级：

- 可导入。
- 可导入但有警告。
- 不可导入。

### 4.3 确认导入

确认导入前，用户应能选择：

- 导入为新故事包。
- 覆盖已有故事包，不建议第一版默认提供。
- 复制资源到资源库。
- 保留原 ID 或重新生成内部 ID。

第一版建议只支持「导入为新故事包」，降低风险。

## 5. 导出故事包

导出流程建议：

```text
选择故事包 -> 运行校验 -> 查看导出摘要 -> 导出文件
```

导出摘要展示：

- 故事包名称。
- 版本。
- 卡片数量。
- 角色数量。
- 资源数量。
- 章节结束配置。
- 起点卡。
- 校验结果。

如果存在错误：

- 禁止导出正式包。
- 可以允许导出调试包，但要明确标记。

如果存在警告：

- 可以导出。
- 需要展示警告列表。

## 6. 导出格式

第一版导出格式应服务于引擎导入。

建议区分：

- 正式故事包：给话机 / 引擎使用。
- 调试故事包：带更多调试元数据。
- 源工程包：带编辑器布局、节点坐标、浮窗偏好。

如果第一版只做一种格式，建议先做「源工程包 + 引擎可读取内容」的组合包，但 UI 上仍应说清楚导出用途。

## 7. 版本与兼容性

导入导出界面必须关注版本兼容。

需要展示：

- 当前 Studio 支持的 schema 版本。
- 当前故事包 schema 版本。
- 是否需要迁移。
- 是否存在不支持字段。

文案示例：

```text
这个故事包来自较早版本，可以迁移后导入。
迁移不会修改原文件。
```

不要直接显示一堆 Zod 错误作为主界面。

## 8. 资源处理

导入故事包时可能包含资源引用。

第一版需要至少表达：

- 资源完整。
- 缺少资源。
- 资源路径无法访问。
- 资源格式不支持。
- 资源已存在。

处理策略：

- 缺资源不一定阻止导入，但必须阻止正式导出。
- 资源冲突时优先保留已有资源，并提示用户处理。
- 不要在导入时自动覆盖资源。

## 9. UI 形态

创建、导入、导出不适合做成大表单页。

推荐形态：

- 从首页或故事包列表打开任务浮层。
- 复杂预检结果可以进入专门的检查页。
- 主要操作按钮固定在底部或右下角。
- 结果反馈明确：成功、警告、失败。

## 10. 验收标准

第一版至少满足：

- 新建故事包时无需填写 ID。
- 新建后直接进入编辑器。
- 导入前有预检。
- 导入不会直接覆盖已有故事包。
- 导出前有校验。
- 有错误时禁止正式导出。
- 警告信息能被用户理解。
- 资源缺失能被明确提示。
- 版本不兼容能被明确提示。

## 11. 避免事项

- 不要把新建故事包做成十几个字段的大表单。
- 不要要求用户填写 packageId。
- 不要导入后才告诉用户 schema 错了。
- 不要静默覆盖已有故事包。
- 不要把导出按钮放得到处都是。
- 不要把校验错误原样甩给用户。

## 12. 一句话总结

创建要快，导入要先检查，导出要先校验；用户负责故事内容，系统负责 ID、结构和兼容性。
