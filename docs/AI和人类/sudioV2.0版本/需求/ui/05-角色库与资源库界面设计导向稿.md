# 用户管理、角色库与资源库界面设计导向稿

> 版本定位：本文定义 Studio V2.0 用户管理、角色库与资源库的设计方向。  
> 用户管理服务调试器，角色库和资源库服务故事编辑器；它们都是支撑能力，不应喧宾夺主。

## 1. 三类支撑对象的定位

用户管理用于维护测试用户档案。这里的用户不是故事角色，而是调试器里模拟「真实来电用户」的档案。

角色库用于管理故事创作中可被 CallCard 归属、挂卡和呼叫的角色。

资源库用于管理被故事包引用的素材，例如 WAV、背景音乐、图片、文本资料等。

它们共同服务于 Studio V2.0：

- 用户管理回答「用哪个用户档案来跑调试」。
- 角色库回答「这张卡属于谁、要挂给谁、谁来打电话」。
- 资源库回答「这张过场卡播放什么、这段剧情引用什么资源」。

## 2. 用户管理页面

用户管理页是正式页面，但它的定位要克制：用于调试，不是玩家社区管理后台。

页面布局：

```text
顶部：搜索 / 新建用户档案
左侧：用户档案列表
右侧：用户档案详情
底部或侧边：最近调试记录
```

用户档案列表展示：

- 昵称。
- 头像，可选。
- 最近调试时间。
- 关联调试记录数量。
- 当前状态：普通 / 常用 / 归档。

用户档案详情建议包含：

- 基本信息：昵称、头像、备注。
- 调试偏好：默认呼入角色、默认故事包、默认是否重置状态。
- 用户背景：可选，用于调试时模拟用户已知信息。
- 最近调试：最近跑过哪些故事包和卡片。
- 高级信息：userId，只读展示。

新建用户档案时：

- 用户填写昵称、头像、备注。
- 系统自动生成 userId。
- 头像可以上传、从资源库选择，或使用默认头像。

第一版不需要：

- 登录账号管理。
- 权限系统。
- 多真实用户运营后台。
- 复杂用户画像。
- 真实通话记录管理。

用户管理页的核心价值是让调试器可以选择不同测试用户，而不是管理产品用户。

## 3. 角色库页面

角色库页面可以是独立页面，也可以从编辑器中以浮窗打开。

第一版建议：

- 首页导航中有独立角色库页面。
- 故事编辑器内有轻量角色浮窗，用于选择和引用。

页面布局：

```text
顶部：搜索 / 新建角色 / 导入
左侧：角色列表
右侧：角色详情
底部或侧边：引用情况 / 关联故事包
```

角色库可以使用列表布局，不需要画布。

## 4. 角色列表

角色列表展示：

- 头像。
- 显示名。
- 角色类型。
- 当前被多少故事包引用。
- 默认自由通话卡状态。
- 最近修改时间。

不要展示：

- agentId 作为主字段。
- prompt 文件路径。
- 底层 runtime 状态。

## 5. 角色详情

角色详情建议分组：

- 基本信息：显示名、头像、简介。
- 电话身份：号码 / 呼叫名 / 默认可拨打状态。
- 默认自由通话：默认 FreeCard 或自由通话策略。
- 人设资料：公开人设、内部设定、关系标签。
- 工具能力：默认可用 FC 能力。
- 引用情况：被哪些故事包、哪些 CallCard 使用。
- 高级信息：agentId、原始 JSON、schema。

第一版可以先做基本信息、默认自由通话、引用情况。

表单编排：角色 / 用户 / 资源详情与创建弹窗统一走 Studio V2 **AutoForm**（`items[]` + ComsMap + 按 name 自动绑；`comProps` / `children` 双逃生口）。不再使用已删除的 FormSchemaRenderer / `kind` 双轨配置。字段语义以引擎 `CharacterDef` 与本轮细化需求为准；记忆区只读，不写角色 JSON。

## 6. 新建角色

新建角色时用户填写：

- 显示名。
- 头像，可选。
- 简介。
- 默认自由通话说明。

系统生成：

- agentId。
- 默认 FreeCard ID，如需要。

不要求用户填写 ID。

### 6.1 角色头像上传

添加角色时必须支持头像设置。

头像来源：

- 上传本地图片。
- 从资源库选择图片。
- 使用默认头像。
- 后续可扩展 AI 生成头像，但第一版不是必须。

头像上传交互：

- 支持 PNG / JPG / WebP。
- 上传后进入裁剪框。
- 裁剪比例默认 1:1。
- 生成小尺寸预览。
- 保存后写入资源库或角色专属资源引用。

头像显示位置：

- 角色库列表。
- 角色详情。
- 故事编辑器左侧角色锚点。
- CallCard 属性浮窗中的所属角色选择。
- 调试器角色状态区。

要求：

- 头像不是内部 ID 的替代品，只是视觉识别。
- 上传失败要给出人话提示。
- 没有头像时使用稳定默认头像，不允许空白破图。
- 删除或替换头像时要保持角色引用不丢失。

### 6.2 头像托管与引擎边界

用户头像和角色头像本质上都是资源库资产，不是引擎能力。

Studio / Next.js 后端负责：

- 接收头像上传。
- 执行裁剪或缩略图生成。
- 保存图片文件。
- 生成资源 `assetId`。
- 在用户档案或角色档案中写入 `avatarAssetId`。
- 在 UI 中读取并展示头像。
- 导入 / 导出故事包时打包头像资源。

引擎只负责：

- 读取用户或角色配置中的头像资源引用。
- 校验 `avatarAssetId` 指向的资源是否存在。
- 在内容包中保留头像引用。
- 将头像引用提供给需要展示的壳层或 Studio。

引擎不负责：

- 上传图片。
- 裁剪图片。
- 生成缩略图。
- 托管静态图片。
- 处理图片格式转换。
- 决定头像展示样式。

推荐数据关系：

```ts
role: {
  id: "role_xxx",
  name: "张老板",
  avatarAssetId: "asset_avatar_abc"
}

userProfile: {
  id: "user_xxx",
  name: "测试用户A",
  avatarAssetId: "asset_avatar_user_abc"
}

asset: {
  id: "asset_avatar_abc",
  type: "image",
  usage: "avatar",
  fileName: "zhang-boss.webp",
  mimeType: "image/webp",
  path: "assets/images/zhang-boss.webp"
}
```

第一版建议：

- Next.js 后端保存头像到 Studio 工程目录的 `assets/images/`。
- 资源库登记一条 image asset。
- 用户或角色只保存 `avatarAssetId`。
- 导出故事包时将 `assets/images/` 一起打包。
- 引擎导入或读取时只校验引用，不处理图片文件本身。

如果未来话机壳层需要显示头像，壳层应从内容包读取头像资源并展示；引擎只提供「这个角色头像资源是哪一个」。

## 7. 编辑器中的角色浮窗

故事编辑器内的角色浮窗不是完整角色库。

用途：

- 快速查看本故事包使用的角色。
- 将 CallCard 归属到某角色。
- 给出口动作选择挂卡目标角色。
- 查看角色当前挂卡数量和状态。

视觉形态：

- 轻量浮窗。
- 可搜索。
- 可拖动。
- 可关闭。
- 不长期占据画布。

## 8. 资源库页面

资源库页面用于管理素材。

第一版资源类型：

- WAV 音频。
- 背景音乐。
- 图片。
- 文本资料。
- 其它文件。

布局建议：

```text
顶部：搜索 / 上传 / 筛选
左侧：类型筛选
中部：资源列表
右侧：资源详情 / 引用情况
```

资源库可以是列表或紧凑卡片，不建议做成媒体瀑布流。

## 9. 资源列表

资源列表字段：

- 资源名。
- 类型。
- 时长或大小。
- 格式。
- 引用数量。
- 最近修改时间。
- 校验状态。

音频资源需要展示：

- 时长。
- 格式。
- 是否可被过场播放卡使用。
- 是否缺失本地文件。

## 10. 资源详情

资源详情建议包含：

- 资源预览。
- 文件信息。
- 引用情况。
- 可用性校验。
- 替换资源。
- 删除或归档。

音频资源应支持：

- 播放预览。
- 时长展示。
- 波形可以后续考虑，第一版不是必须。

注意：引擎不负责真正播放 WAV，资源库只管理剧情资源引用和壳层可执行意图。

## 11. 编辑器中的资源浮窗

资源在故事编辑器中应以浮窗使用。

用途：

- 给过场播放卡选择 WAV。
- 给剧情卡引用资料。
- 查看资源缺失警告。

要求：

- 默认可收起。
- 不长期占据画布。
- 支持搜索和类型筛选。
- 选择资源后回填到当前卡片或出口动作。

## 12. 引用关系

用户、角色和资源都必须能看见引用关系。

引用关系用于：

- 删除前确认。
- 导出前检查。
- 调试问题定位。
- 迁移和导入时处理冲突。

引用展示要用业务语言：

```text
被「张老板回拨」过场播放卡引用
被「聊崩了」出口动作引用
被「内存条事件」调试记录使用
```

不要只显示路径或 ID。

## 13. 删除与归档

第一版建议优先做归档，不鼓励直接删除。

删除用户、角色或资源前必须提示：

- 被哪些故事包引用。
- 被哪些 CallCard 使用。
- 被哪些调试记录使用。
- 删除后会造成哪些校验错误。

## 14. 验收标准

第一版至少满足：

- 可以创建用户档案，ID 自动生成。
- 可以为用户档案上传或选择头像。
- 调试器可以选择用户档案。
- 可以创建角色，ID 自动生成。
- 可以为角色上传或选择头像。
- 可以查看角色被哪些故事包或卡片引用。
- 可以在编辑器中选择角色作为 CallCard 所属角色。
- 故事编辑器角色锚点能显示角色头像。
- 可以上传或登记音频资源。
- 可以在过场播放卡中选择资源。
- 可以看到资源缺失状态。
- 可以看到资源被哪些卡片引用。
- 资源库可以作为浮窗在编辑器内打开。

## 15. 避免事项

- 不要把用户管理做成真实账号权限系统。
- 不要把角色库做成 prompt 文件编辑器。
- 不要把资源库做成网盘。
- 不要让用户手填 userId、agentId 或 assetId。
- 不要让角色没有头像时显示破图。
- 不要让资源浮窗挤压画布。
- 不要删除被引用资源却没有提示。
- 不要把 WAV 播放实现责任放到引擎 UI 上。

## 16. 一句话总结

用户管理服务调试器，角色库和资源库服务故事编辑器；独立页面负责管理，编辑器和调试器里的浮窗负责快速引用。
