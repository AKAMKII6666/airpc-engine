# Studio V2 前端工程规则与质量门禁需求

## 1. 文档目的

Studio V2 不只需要新的界面设计，也需要从第一天建立可执行的工程规则。

旧 UI 的问题已经证明，仅在文档里写“代码要清晰”“适当拆分”“需要注释”没有约束力。页面和 Hook 会继续变长，业务判断会重新进入组件，关键状态转换没有说明，最终又只能靠阅读者猜测代码意图。

因此本需求规定两件事：

1. Studio V2 编码时必须遵守的结构与注释规则。
2. 后续质量门禁脚本必须自动检查的范围、结果和接入方式。

本阶段只定义门禁需求，不在本需求包内实现脚本。

目录落点的长期真源见 [09-Studio-V2前端目录结构与落点规则.md](./09-Studio-V2前端目录结构与落点规则.md)。本文只描述质量门禁能力；具体文件应该落到哪里，以 09 为准。

## 2. 基本原则

- 门禁用于阻止明确的结构退化，不替代代码审查。
- 能由类型、命名和拆分表达清楚的内容，不要求用注释重复一遍。
- 注释重点解释意图、约束、时序、所有权和副作用。
- 新 Studio 从零执行严格门禁；旧 Studio 的历史问题不得成为 V2 的豁免理由。
- 所有门禁必须能在本地和 CI 中运行，并给出可定位、可修复的错误。
- 禁止通过压缩代码、删除注释或大范围忽略目录来“通过”门禁。

## 3. 代码结构规则

### 3.1 页面与组件职责

- `page.tsx` 只负责路由入口、页面壳、首屏装配和错误/加载边界。
- 页面不得直接实现完整的保存、导入导出、调试运行或画布命令流程。
- 展示组件不得直接拼 BFF URL、读写持久化真源或复制引擎规则。
- 编辑器命令、状态转换、请求编排、格式转换必须进入各自的 command、domain、service 或 shell/bis 层。
- 一个组件只承担一个可命名的界面职责；属性窗、出口动作、角色锚点、校验窗等应独立演进。

### 3.2 文件与函数规模

第一版建议门禁阈值如下：

| 检查对象 | 告警线 | 硬上限 | 处理要求 |
|---|---:|---:|---|
| `page.tsx` | 120 行 | 200 行 | 超限必须拆出 feature 或页面组件 |
| 普通 UI 组件 | 250 行 | 400 行 | 超限按展示区域或交互职责拆分 |
| Hook / bis / shell / command 文件 | 250 行 | 400 行 | 超限按用例或状态域拆分 |
| 其他手写 TS/TSX 文件 | 350 行 | 500 行 | 超限必须说明并拆分职责 |
| 单个函数或 Hook | 60 行 | 100 行 | 超限拆出纯函数、命令或步骤对象 |
| 圈复杂度 | 10 | 15 | 超限重构条件分支或状态机表达 |

统计时忽略空行，但不忽略注释。注释是代码可维护性的一部分，不能为了行数门禁被删除。

以下文件可以采用独立阈值，但必须在门禁配置中显式登记：

- 自动生成文件。
- schema 生成产物。
- 只包含静态 token 或映射表的文件。
- 测试夹具与快照。

业务源码不允许仅凭“这个文件比较特殊”永久豁免。

### 3.3 依赖边界（Client / Server 端口隔离）

V2 必须延续 Browser → Next 门面 → EngineHost / 引擎的单向边界，并按以下硬规则隔离端口：

1. **交互唯一通道**：Client 与 Server/引擎之间禁止任何模块级 `import`（含 `import type`）。唯一合法互通是 XHR/fetch（经 ajaxHelper）↔ Next API ↔ 引擎。
2. **类型不共享**：同一概念若前后端都要用，必须写两套类型（FE `typeFiles/` 与引擎/Server schema），哪怕字段一致。禁止 Client `import type` 引擎。存量违规必须立刻清零，无 baseline。
3. **功能不交叉**：引擎/Server 的函数、Zod Schema、校验器、常量表，Client 不得 import。需要同构逻辑时在 Client 复制轻量镜像（注释标明与引擎同构、不以 import 同步）。独立性优先于 DRY。
4. **反向同样**：`app/api/**`、`src/utils/server/**`、`*.server.ts(x)` 不得 import Client 区路径（`src/pageComponents/**`、`src/bis/**`、`src/stores/**`、`src/commonUiComponents/**`、`src/utils/ajaxProxy/**`、`typeFiles/**` 等）。

补充约束：

- 不得保存 LLM Key，不得直接修改 `data/**`。
- **仅** Server 路径允许 `import from "@airpc/rpg-engine"`；禁止深挖引擎内部路径。
- 允许复制 DTO 形状与本地预检；调试器/编辑器不得复刻 ExitSelector、WET 或 EffectExecutor 的裁决语义。
- 分区与检测由 `structure-gate-config.json` 配置，`check:studio-structure` 硬失败（`STUDIO-STRUCT-005` / `STUDIO-STRUCT-020`）。

### 3.3.1 客户端分层依赖（store · shell · bis）

对齐 [技术设计 21](../../../技术设计文档/21-Studio客户端分层.md)：

| ID | 约束 |
|----|------|
| `STUDIO-STRUCT-021` | `pageComponents/**`、`commonUiComponents/**` 禁止 import `stores/**`、`utils/ajaxProxy/**` |
| `STUDIO-STRUCT-022` | `stores/**` 禁止 import `bis/**`、`ajaxProxy/**`、`next/navigation` |
| `STUDIO-STRUCT-023` | 非 `*.shell.bis.ts` 且非 `*Flow.ts` 的 bis 禁止 `next/navigation` |
| `STUDIO-STRUCT-024` | bis 禁止 value import `pageComponents/**`、`commonUiComponents/**` |

存量违规以 `scripts/studio-quality/studio-v2-layering-baseline.json` 按文件内容哈希冻结；**触碰即清**。禁止宽目录永久 allowlist。新文件一律硬失败。

### 3.4 Studio V2 固定目录

Studio V2 业务代码只认以下顶层结构：

```text
app/
src/
typeFiles/
tests/
```

`app/` 只做路由入口，`src/pageComponents/` 放页面组件，`src/bis/` 放页面和组件编排，`src/stores/` 放前端状态，`src/utils/` 放公共工具，`typeFiles/` 放类型契约。

新代码禁止继续使用旧式 `features/`、`domain/`、`services/`、`commands/`、`store/` 作为顶层业务目录。迁移时必须同步更新 import 并删除旧路径，禁止保留兼容转发壳。

## 4. 注释规则

### 4.1 必须有注释的代码

以下新增或修改内容必须具备就近的意图注释或 JSDoc：

- 对外导出的领域模型、端口、服务接口和编辑器命令。
- CallCard、Exit、Effect、WET、调度、章节切换等关键状态转换。
- 跨 client/server/engine 边界的数据适配与单位转换。
- 存在时序要求、幂等要求、重试语义、事务边界或异步副作用的函数。
- 兼容旧 schema、降级、迁移和临时绕过逻辑。
- 仅从函数名无法判断原因的复杂算法、布局修正和画布坐标转换。
- 导出领域 `interface` / `type` 中存在特殊语义的字段。

字段级说明至少应覆盖适用的内容：

- 数值单位，例如毫秒、秒、画布坐标或逻辑时钟。
- `null`、缺省值和空数组分别代表什么。
- 字段由谁生成、由谁修改、是否持久化。
- 字段之间的互斥、前置条件和生命周期。
- 是否只用于 UI 投影，是否属于内容真源。

### 4.2 不要求注释的代码

- 语义清楚的局部变量和小型纯函数。
- 简单的 JSX 展示、样式映射和事件转发。
- 已经被类型与命名完整表达的普通 DTO 字段。
- 框架固定样板。

门禁不得要求每个函数、每个变量或固定比例的代码都带注释。

### 4.3 注释质量

合格注释应说明以下至少一项：

- 为什么这样做。
- 必须保持什么不变量。
- 什么时候产生副作用。
- 数据或状态由谁拥有。
- 哪个失败语义必须被调用方处理。
- 为什么不能使用看起来更直接的实现。

以下内容不能视为有效注释：

- `// 设置状态`、`// 调用接口`、`// 遍历数组` 等复述代码。
- 只写文件名或函数名的文件头。
- 与实际行为不一致的过期注释。
- 为达到注释密度而批量生成的模板句。

### 4.4 绕过注释

以下指令必须带有就近原因：

- `eslint-disable` / `eslint-disable-next-line`
- `@ts-ignore`
- `@ts-expect-error`
- 门禁 ignore / allowlist 项
- 临时兼容分支与 TODO

临时项还必须说明退出条件。建议格式：

```ts
// 原因：第三方节点类型未暴露 viewport 字段；退出条件：升级到包含该类型的版本后删除。
```

## 5. 门禁脚本需求

### 5.1 脚本组成

后续至少实现以下命令：

```text
npm run lint
npm run check:comments
npm run check:studio-structure
npm run test:studio
npm run quality:studio
```

其中：

- `check:comments` 检查关键声明注释、字段级说明和绕过原因。
- `check:studio-structure` 检查文件/函数规模、复杂度、目录与依赖边界。
- `quality:studio` 汇总 lint、类型检查、注释、结构和测试，作为 CI 的唯一 Studio 质量入口。

实际命令名可以在工程落地时调整，但能力不得减少。

### 5.2 注释门禁实现要求

注释门禁必须基于 TypeScript AST 或 ESLint AST，不得只用正则扫描源码。

脚本至少检查：

1. 配置范围内的公共领域声明是否具备说明。
2. 严格契约目录中的导出字段是否具备字段级说明。
3. `eslint-disable`、TypeScript 抑制指令是否带原因。
4. 临时 allowlist 是否包含原因、责任域和退出条件。
5. 新增复杂函数是否缺少意图/副作用说明。

脚本不负责判断所有注释是否优美，但应识别明显无效内容，例如只重复声明名、空 JSDoc、统一模板占位词。

V2 根目录确定时，门禁配置必须显式声明“严格契约目录”，例如领域 contract/schema、client-server DTO 和公开 command 类型。严格目录中的导出字段全部检查；普通 UI props 和内部 DTO 不进入字段级强制范围。这样脚本有稳定判定依据，也避免全仓机械补注释。

### 5.3 复杂度门禁实现要求

结构脚本或 ESLint 配置至少检查：

- 文件有效行数。
- 函数有效行数。
- 圈复杂度。
- `page.tsx` 是否超出专用阈值。
- UI 组件是否直接使用禁用依赖或发起裸请求。
- 非 Server 路径是否 import `@airpc/rpg-engine`（含 `import type`；`STUDIO-STRUCT-005`）。
- Server 路径是否倒引 Client 区模块（`STUDIO-STRUCT-020`）。
- V2 是否深挖引擎内部路径（`STUDIO-STRUCT-006`）。
- 测试是否错误地与业务文件平放。
- 新增目录是否违反职责聚类规则。
- 是否出现旧式顶层业务目录、兼容转发壳、父子同名重复入口。

其中“一个组件承担多个业务域”等语义问题仍由代码审查判定，不伪装成机器能够完全理解。

### 5.4 检查范围

- V2 工程根目录一旦确定，必须作为全量强制范围写入配置。
- 本地开发可以提供“仅检查本次变更”模式，用于快速反馈。
- CI 必须全量检查 V2，不得只检查 diff。
- 生成目录、构建产物、第三方代码和快照必须通过明确配置排除。
- 旧 Studio 可以先使用基线模式，但 V2 不允许建立大规模历史基线。

### 5.5 输出格式

每个问题必须输出：

```text
规则编号  文件:行:列  问题说明  修复建议
```

示例：

```text
STUDIO-COMMENT-003 apps/studio-v2/domain/callCard.ts:42:3
出口动作字段缺少单位/生命周期说明；请补充该字段由谁生成以及何时消费。
```

门禁失败必须返回非零退出码。CI 还应支持输出结构化 JSON 报告，供后续审查工具和编辑器读取。

### 5.6 豁免机制

允许极少量、可审计的豁免，但必须满足：

- 集中登记，不允许散落任意 ignore glob。
- 包含规则编号、精确文件或声明、原因、责任域、创建日期和退出条件。
- 禁止 `apps/**`、`features/**` 等大目录级豁免。
- 新增豁免必须在代码审查中单独显示。
- 已满足退出条件的豁免应由门禁报告为失败，推动删除。

## 6. 测试要求

门禁脚本本身也必须有测试，至少覆盖：

- 合格的公共契约通过。
- 缺失关键注释失败。
- 复述式占位注释失败。
- 带原因的 TypeScript 抑制通过，无原因的失败。
- 文件、函数和复杂度阈值的边界值。
- 生成文件排除。
- 精确豁免生效，大范围豁免被拒绝。
- Windows/macOS/Linux 路径输出保持可定位。

测试夹具应独立放置，不得混入 Studio 业务源码。

## 7. CI 与开发流程

推荐流程：

```text
开发时：编辑器 ESLint 即时提示
提交前：quality:studio
CI：全量 quality:studio
合并：任何硬门禁失败都阻断
```

告警线用于提示即将恶化的文件；硬上限、依赖边界、无理由抑制和必需注释缺失必须阻断。

第一版脚本落地时，应先对 V2 空骨架和示例 feature 建立通过基线，再开始批量开发页面。这样规则从第一份正式代码就生效。

## 8. 验收标准

本门禁能力完成时至少满足：

- 根目录存在统一的 Studio 质量命令。
- V2 所有手写 TS/TSX 都进入全量检查范围。
- 关键领域契约缺少意图说明时 CI 会失败。
- 无原因的 lint/TypeScript 抑制会失败。
- 超过硬上限的页面、组件、Hook 和函数会失败。
- client/Next/engine 依赖越界会失败。
- 错误能定位到具体文件与声明，并给出修复方向。
- 门禁脚本自身有自动化测试。
- 不存在大目录永久豁免。
- `quality:studio` 通过后，才允许把 Studio V2 任务标记为完成。
