# 新版故事编辑器界面 UI 设计向导稿

> 版本定位：这是 Studio V2 故事编辑器的界面设计向导稿，用来对齐产品形态、信息层级、交互规则和引擎语义映射。  
> 本文不是代码实现方案，也不是最终视觉规范；后续可以在此基础上继续拆成视觉规范、组件规范、交互验收清单。

## 1. 为什么需要这份向导

新版故事编辑器不是传统后台管理系统，也不是 JSON 配置页。它要解决的是「剧情电话链条」的可视化创作问题：

- 一张 CallCard 代表一通电话的剧情驱动。
- 一通电话可以由多个上一张卡进入。
- 一通电话可以有多个出口。
- 一个出口不只是连接下一张卡，还可以执行一组动作：给多个角色挂卡、卸载卡、延迟外呼、设置世界状态、结束章节等。
- 角色归属、剧情流转、调试状态、校验问题需要同时可见，但不能互相抢视觉。

旧 UI 的主要问题是把编辑器做成了固定三栏表单和 JSON 控制台，画布空间被挤压，卡片、角色、出口和状态变化之间的关系无法自然表达。新版 UI 应该从一开始就按「自由画布 + 蓝图节点 + 轻量浮窗」来设计。

## 2. 核心设计结论

新版故事编辑器的基本形态是：

**全屏自由画布 + 画布角色锚点 + 底部工具条 + 按需出现的浮动功能窗口。**

必须坚持以下原则：

1. 画布是主角，故事链条必须铺满整个编辑空间。
2. 除角色锚点外，功能区都应是可打开、可关闭、可拖动、可收起的绝对定位浮窗。
3. 不默认使用泳道。泳道看起来整齐，但会牺牲自由布局，且与多父节点、多出口、多角色挂卡的剧情结构不匹配。
4. CallCard 节点必须像蓝图节点一样表达入口、出口和动作，而不是像普通卡片一样只展示字段。
5. 出口是一个一等触点。一个出口触点可以连接多个动作目标。
6. 角色归属线和剧情出口线必须分层显示，不能混成一种线。
7. 用户不应该手填任何 ID。所有故事、角色、卡片、出口、动作 ID 都应创建时自动生成。
8. 主界面只展示人能理解的业务词，不把 `cardId`、`packageId`、`condition.op` 这类底层字段暴露成主要操作对象。

## 3. 默认界面形态

### 3.1 顶部悬浮工具栏

顶部工具栏应该轻，像画布工具的状态条，不像后台系统 Header。

建议包含：

- 项目 / 故事包名称。
- 当前章节名称。
- 保存状态。
- 校验状态。
- 运行调试入口。
- 导出入口。
- 搜索入口。
- 视图模式入口。

顶部工具栏不承载复杂配置。复杂配置应该进入浮窗或专门页面。

### 3.2 画布角色锚点节点（唯一入口）

> **细化修改 4 定稿：** 不再保留左侧固定角色轨，也不在底栏提供「角色浮窗」。本包角色的新增 / 编辑 / 归属导航 **只在画布角色锚点节点** 完成。

角色锚点是画布上的一等节点（可连线的 CharacterAnchor），不是侧栏列表，也不是 docker 浮窗。

每个角色锚点建议展示：

- 头像。
- 角色名。
- 当前状态：自由通话 / 有待呼入卡 / 有待外呼卡 / 延迟外呼中 / 被锁定。
- 当前挂卡数量。
- 当前选中卡是否属于该角色的高亮状态。

交互规则：

- 点击角色锚点：选中该节点；可高亮该角色相关的所有 CallCard 和角色归属线。
- 双击或打开编辑：复用 `/characters` 同一套 FormModal / AutoForm（A1），可写 `data/characters`；禁止在编辑器内另起平行角色表单。
- 新增角色：在画布上创建锚点节点，同样走角色库新建表单；落盘后同步加画布节点。
- 拖动角色锚点不改变角色本体，只改变画布中角色归属线的锚点位置或视觉偏好。
- CallCard 顶部 `role` 口连到角色锚点，表达归属。

角色锚点的目标：帮助用户在大画布中导航剧情归属，并作为本包角色资料的唯一编辑入口。

### 3.3 全屏自由画布

画布应占据界面的绝大部分空间。

基础要求：

- 支持平移、缩放、框选、拖拽节点。
- 支持无限画布或大尺寸画布。
- 支持吸附到轻量网格，但不能强制对齐到泳道。
- 支持一键整理，但整理结果不能破坏用户手动布局。
- 支持局部聚焦：选中卡片时弱化无关节点和连线。

画布上主要对象：

- CallCard 节点。
- Chapter Start / Chapter End 节点。
- Action 节点。
- State 节点。
- 注释分组。
- 调试标记。

### 3.4 底部工具条

底部工具条用于高频创建和视图操作。

建议包含：

- 新建 CallCard。
- 新建过场播放卡。
- 新建自由通话卡。
- 新建延迟外呼卡。
- 新建章节结束节点。
- 新建动作节点。
- 新建注释分组。
- 连线模式。
- 框选模式。
- 运行当前位置。
- 缩放 / 适配视图。

底部工具条要小，不要变成第二个菜单栏。

### 3.5 浮动窗口系统

角色锚点已在画布节点上，不另做侧栏/底栏角色列表浮窗。其余功能区默认都是浮动窗口。

常见浮窗：

- 卡片模板窗口。
- 卡片属性窗口。
- 出口动作窗口。
- 包配置窗口（只读，底栏 docker 打开）。
- 资源窗口（可 CRUD，复用 `/assets`）。
- 校验问题窗口。
- 调试器窗口。
- 最近变更窗口。
- 小地图。
- JSON 高级视图窗口。

浮窗规则：

- 默认不抢占画布宽度。
- 可以拖动、关闭、最小化、固定。
- 属性窗口默认出现在被选中卡片或出口触点附近。
- 用户可以把某个浮窗钉住，但钉住状态不应成为默认布局。
- 所有浮窗应有「收起全部」和「恢复默认布局」能力。

## 4. CallCard 节点设计

### 4.1 三类连接面

CallCard 节点有三类连接面：

```text
                 顶部：角色归属连接口
                         │
          ┌─────────────────────────┐
左侧入口  │        CallCard          │  右侧出口触点
Parent ──▶│   标题 / 类型 / 目标      │──▶ Exit A
Parent ──▶│   状态 / 摘要 / 风险      │──▶ Exit B
          └─────────────────────────┘──▶ Exit C
```

- 顶部连接口：连接所属角色锚点。
- 左侧连接口：接收上一张卡或章节开始节点，可以有多个 parent。
- 右侧连接口：出口触点，可以有多个 exit。

这三个连接面必须在视觉上区分清楚，避免用户误以为「属于某角色」和「剧情流向某角色」是一回事。

### 4.2 卡片信息层级

卡片上只展示最核心的信息，字段要压缩。

建议主卡片展示：

- 卡片标题。
- 卡片类型：剧情通话 / 自由通话 / 延迟外呼 / 过场播放 / 章节结束。
- 所属角色。
- 本轮目标一句话。
- 入口状态：可呼入 / 待外呼 / 可提前呼入 / 仅播放。
- 出口数量。
- 校验状态。

不建议直接展示：

- 长背景故事。
- 完整提示词。
- 原始 JSON。
- UUID。
- 大量底层字段。

这些内容应在属性浮窗中查看和编辑。

### 4.3 卡片视觉状态

卡片需要有清晰状态，但不能过度装饰。

建议状态：

- 草稿：中性边框。
- 已校验：轻量成功标记。
- 有警告：黄色角标。
- 有错误：红色角标。
- 调试经过：细高亮描边。
- 当前运行中：明确但克制的运行态。
- 被选中：强化边框、显示附近属性浮窗。
- 被聚焦角色关联：正常显示。
- 非关联：降低透明度。

卡片样式应该更像「剧情节点」，而不是后台信息卡。节点要紧凑、可大量铺开、便于看线。

## 5. 出口触点与动作列表

### 5.1 出口不是一条线

新版编辑器里，出口不是简单的「连到下一张卡」。

出口应该是一个一等触点：

```text
CallCard: 张老板回拨
  Exit: 聊崩了
    ├─ 挂卡：小雨 / 安抚用户
    ├─ 挂卡：强叔叔 / 调查内存条
    ├─ 挂卡：澜星姐姐 / 解释失败
    └─ 设置世界状态：memory_quote_failed = true
```

也就是说，一个出口触点后面是一组动作列表。

### 5.2 出口动作类型

第一版至少应支持这些动作类型的设计表达：

- 挂卡到角色：给某个角色挂一张 CallCard。
- 卸载角色卡：清掉某个角色或全部角色的剧情卡。
- 延迟外呼：在指定延迟后由某角色主动打给用户。
- 取消延迟外呼：取消某个已安排的延迟外呼。
- 设置世界状态：写入剧情变量或世界状态。
- 更新角色知识：给角色追加已知事实。
- 结束章节：进入章节结束点，并按结束点配置安排下一章节。
- 播放提示 / 留言：安排壳层播放资源或创建用户可感知提示。
- 自由通话回退：让目标角色回到自由通话卡。

注意：媒体播放本身不由引擎执行，编辑器只表达剧情资源和壳层动作意图。

### 5.3 出口动作窗口

点击某个出口触点时，应在触点旁边打开轻量浮窗。

窗口内容建议：

- 出口名称。
- 出口判定说明。
- 成功 / 失败 / 兜底类型。
- 动作列表。
- 每个动作的目标角色、目标卡片、延迟、是否 critical。
- 新增动作按钮。
- 动作排序。
- 动作校验结果。

不要把出口动作塞进右侧固定大面板。出口是局部结构，属性窗口应该贴近出口触点。

## 6. 连线系统

连线必须系统化，否则故事稍微复杂就会糊成一团。

### 6.1 角色归属线

用途：表达 CallCard 属于哪个角色。

规则：

- 从 CallCard 顶部连接口连到画布角色锚点。
- 默认使用半透明、浅色、虚线、细贝塞尔曲线。
- 默认透明度低，不抢剧情出口线视觉。
- 选中卡片时，该卡片的角色归属线变实、透明度提升。
- 点击角色锚点时，该角色所有归属线提高透明度。
- 角色归属线不显示箭头，避免被误解成剧情流转方向。

### 6.2 剧情流转线

用途：表达从一个出口进入另一张 CallCard。

规则：

- 从右侧出口触点连到目标 CallCard 左侧入口。
- 使用实线和箭头。
- 颜色强于角色归属线。
- 线条上可显示短标签，例如「用户去打」「无应答超时」「聊崩了」。
- 多个 parent 进入同一张卡时，目标卡左侧可以显示多个入口点或聚合入口点。

### 6.3 动作线

用途：表达出口触点连接到动作节点或状态节点。

规则：

- 从出口触点连到 Action / State / End 节点。
- 可使用不同于剧情流转线的颜色或线型。
- 多动作出口可在视觉上先聚合为一个动作组，再展开。
- 关键动作可以有更明确的标识。

### 6.4 失败与兜底线

用途：表达失败出口、超时出口、LLM 未命中出口后的兜底策略。

规则：

- 失败线可以使用红色或橙色系，但要克制。
- 兜底线可以使用虚实混合或特殊标签。
- 失败出口仍然是出口触点，仍然可以连接多个动作。

### 6.5 降噪规则

必须提供连线降噪机制：

- 只高亮当前选中节点的一跳关系。
- 支持隐藏角色归属线。
- 支持只看剧情流转线。
- 支持只看某个角色相关线。
- 支持只看错误链路。
- 支持小地图中显示整体结构，主画布按焦点显示细节。

## 7. 关键交互流程

### 7.1 新建 CallCard

流程：

1. 用户从底部工具条或卡片模板浮窗选择卡片类型。
2. 系统自动生成 ID。
3. 用户在画布上点击放置。
4. 属性浮窗在卡片旁边打开。
5. 用户选择所属角色或拖动顶部归属口连接角色锚点。

原则：

- 不要求用户手填 ID。
- 不要求用户先进入复杂表单。
- 新卡片应该先成为画布节点，再逐步补属性。

### 7.2 从出口创建下一张卡

流程：

1. 用户从某张卡右侧出口触点拖出连线。
2. 松开到空白画布。
3. 系统弹出轻量菜单：新建剧情通话 / 新建延迟外呼 / 新建过场播放 / 新建自由通话 / 新建动作。
4. 创建目标节点并自动建立连接。
5. 出口动作列表同步生成对应动作。

这能保证「视觉连线」和「引擎动作」一致。

### 7.3 一个出口挂多张卡

流程：

1. 用户选中出口触点。
2. 出口动作窗口打开。
3. 用户添加多个「挂卡到角色」动作。
4. 画布上从同一出口触点分出多条线，分别连接到不同目标卡。
5. 每条线在目标卡顶部或卡片角标中体现目标角色。

示例：

```text
Exit: 聊崩了
  -> 小雨：安抚用户
  -> 张老板：解释失败
  -> 强叔叔：调查供应商
```

### 7.4 选中卡片

选中卡片后：

- 卡片边框强化。
- 卡片顶部角色归属线变实。
- 所属角色锚点高亮。
- 一跳 parent 和 next 卡片正常显示。
- 无关节点降低透明度。
- 卡片属性浮窗在卡片旁边打开。

属性浮窗应贴近卡片，而不是固定在右侧。

### 7.5 选中角色

选中角色锚点后：

- 高亮该角色所有卡片。
- 高亮该角色归属线。
- 可显示该角色当前剧情队列、自由通话卡、延迟外呼状态。
- 可快速定位到该角色当前挂卡。

### 7.6 调试运行

调试器不应该只是 JSON 展示器。

调试运行时应在画布上直接表达：

- 当前通话卡。
- 当前出口判定。
- WET 输出。
- EffectExecutor 执行结果。
- 哪些角色被挂卡。
- 哪些角色被卸载卡。
- 是否安排延迟外呼。
- 是否进入章节结束点。
- 哪个动作失败，是否 critical。

调试窗口只作为辅助日志，不是主体验。

## 8. 属性浮窗设计

### 8.1 卡片属性浮窗

卡片属性浮窗建议分组：

- 基本信息：标题、类型、所属角色。
- 通话入口：可呼入、待外呼、允许提前呼入、延迟策略。
- 剧情目标：本轮目标、成功条件、失败条件。
- 上下文：背景、前提概要、上一轮信息、不可明说信息。
- 情绪与语气：情绪、关系姿态、语气限制。
- 工具能力：本卡允许的 Function Calling 能力；`toolPolicy.mode=allowlist` 时以**内置工具多选**写入 `allowedToolIds`（中文名 + toolId），禁止自由文本手填 toolId；非 allowlist 时隐藏或禁用该多选。
- 出口概览：出口列表和状态；`exits.effects[].effect` 为**枚举 Select**（中文名），禁止自由文本填 Effect id。
- 高级：原始字段、UUID、JSON 预览。

字段名称应优先中文展示。必要时可在括号中展示原字段名，例如「所属角色（roleId）」。引擎/类型中的枚举字段一律用 Select / Radio / 多选等枚举控件，不出现「枚举却用 TextField」的残留。

### 8.2 出口属性浮窗

出口属性浮窗建议分组：

- 出口名称。
- 触发语义。
- 判定提示。
- 是否兜底。
- 动作列表。
- 失败策略。
- 调试命中历史。

出口属性比卡片属性更局部，窗口要更小、更贴近出口触点。

### 8.3 JSON 高级视图

JSON 可以存在，但只能是高级视图。

要求：

- 使用正式代码编辑器展示，例如 Monaco Editor。
- 支持格式化、折叠、搜索。
- 支持只读 / 可编辑模式切换。
- 编辑 JSON 后必须经过 schema 校验。
- 不允许把原始 JSON textarea 当作主要编辑方式。

## 9. 字段语言与业务词汇

主界面应该使用用户能理解的剧情创作语言。

建议使用：

- 通话卡。
- 自由通话。
- 剧情通话。
- 过场播放。
- 延迟外呼。
- 可提前呼入。
- 当前挂卡。
- 出口。
- 失败兜底。
- 动作列表。
- 挂卡。
- 卸载卡。
- 结束章节。
- 下一章节。
- 世界状态。
- 角色知识。
- WET 结果。
- Effect 执行。

避免在主界面直接使用：

- `cardId`
- `packageId`
- `agentId`
- `condition.op`
- `payload`
- `effect.type`
- `toolPolicy`
- `runtime instance`

底层字段可以存在于高级视图，但不能成为创作入口。

## 10. 与引擎语义的映射

新版 UI 不是纯视觉想象，必须贴合引擎语义。

### 10.1 CallCard 节点

UI 中的一张 CallCard 节点，对应引擎中的一张通话卡定义。

它应承载：

- 入口方式。
- 所属角色。
- 剧情上下文。
- 本轮目标。
- 可用 FC 能力。
- 出口定义。
- 出口动作。

### 10.2 出口动作列表

引擎中一个 exit 可以挂载 effects 数组，因此 UI 中一个出口触点可以设计成动作列表。

这意味着：

- 一个出口可以同时给多个角色挂不同卡。
- 一个出口可以同时设置世界状态。
- 一个出口可以安排延迟外呼。
- 一个出口可以卸载其它角色旧卡。
- 一个出口可以进入章节结束点。

UI 必须把这种「一出口多动作」作为核心能力，而不是后期补丁。

### 10.3 多 parent 进入

一个 CallCard 可以被多个上一张卡的出口指向。

UI 上应支持：

- 左侧多个入口线。
- 入口聚合。
- 查看所有 parent。
- 校验循环和不可达节点。

### 10.4 角色归属与剧情流转分离

CallCard 属于某角色，不等于剧情流向某角色。

因此：

- 顶部角色归属线只表示 belongRole。
- 右侧出口线才表示剧情推进。
- 出口动作中的挂卡才表示「把某张卡挂到某角色身上」。

这是新版 UI 最关键的概念分离之一。

## 11. 需要避免的问题

新版 UI 必须避免旧 UI 的问题：

1. 不要固定三栏布局挤压画布。
2. 不要把属性面板长期固定在右侧。
3. 不要默认展示大段 JSON。
4. 不要让用户手填 ID。
5. 不要中英文混杂做主要字段名。
6. 不要把调试器做成日志和 JSON 展示器。
7. 不要默认使用泳道。
8. 不要让角色归属线和剧情出口线视觉权重一样。
9. 不要让卡片信息过载。
10. 不要让卡片模板窗口长期占据画布空间。
11. 不要让小地图和校验窗口变成大面板。
12. 不要做无法解释引擎状态的装饰性连线。

## 12. 第一版界面验收标准

第一版故事编辑器至少应满足：

- 打开编辑器后，用户第一眼看到的是大画布，而不是表单。
- 角色入口仅为画布角色锚点节点（可新增/编辑，复用角色库表单）；无左侧固定角色轨、无底栏角色浮窗。
- 底栏 docker：资源浮窗 + 包配置浮窗（只读）；无角色 icon。
- 卡片模板默认可收起。
- 点击卡片后，属性浮窗出现在卡片附近；字段对齐 `CallCardDefinition`（mock 阶段可不写盘）。
- 点击出口后，出口动作浮窗出现在出口附近；出口 Handle 按 `exits[]` 动态渲染，悬停 Tooltip 显示名称与概要。
- 卡片顶部可以连接角色归属。
- 卡片左侧支持多个 parent 进入。
- 卡片右侧支持多个出口触点。
- 一个出口触点可以连接多个目标或动作。
- 角色归属线默认足够淡，不抢剧情线。
- 选中卡片后，角色归属线和一跳剧情线清晰可见。
- 校验问题窗口小而克制，不占据主编辑区；校验态仅节点 badge，不进属性 Formik。
- 小地图小而克制，只负责定位。
- JSON 只在高级视图中出现，并使用格式化编辑器。
- 所有 ID 自动生成，主流程不要求手填。
- 选中 `chapter_end` 时，属性区提供下一故事包 / 起点卡 **Select**（非自由文本 ID）。
- 出口 `effects[].effect` 为枚举下拉（中文名）；`toolPolicy.allowlist` 下 `allowedToolIds` 为内置工具多选。

## 13. 已收口的第一版设计决策

这些问题已经形成第一版设计决策：

- 角色锚点在画布上，不另做左侧轨排序 UI。第一版按节点布局表达位置，避免把角色导航做复杂。
- 不做章节区域或剧情分组。一个故事包就是一个章节，先把单章节基础编辑体验打磨好。
- 不保留泳道视图。泳道既不是默认视图，也不作为第一版整理工具。
- 出口动作过多时，不额外拆成独立 Action 节点，也不引入复杂折叠模型。第一版仍以出口动作列表为准，视觉上可以做轻量收起。
- 调试器暂不讨论时间轴回放。调试器会单独设计，不在本 UI 向导中展开。
- 卡片属性浮窗不支持多卡批量编辑。第一版聚焦单卡、单出口的精确编辑。
- 资源管理做成浮动窗口，不做独立页面。
- 暂不做故事包地图。先打磨基础编辑器，后续再考虑总览能力。
- 引擎已知枚举（Effect 名、工具 id、性别/音色等）在 UI 中一律下拉或等价多选，附中文名；禁止自由文本填枚举值（细化修改 5）。

章节结束点的第一版配置方式：

- 直接在章节结束节点或其属性浮窗中配置（会话 mock，不写盘）。
- 用 **Select** 选择下一个故事包（`nextPackageId`），禁止自由文本包 ID。
- 下一章节起点卡（`nextEntryCardId`）从所选故事包的可选卡集合中 **Select**；换包后若不在新集合内则置空或回退默认起点卡。
- 第一张卡应理解为该故事包的起点卡。
- 进入章节结束点时，当前故事包的角色剧情卡应全部清理；在下一个故事包启动前，用户与任意角色通话都走自由通话。

## 14. 一句话总结

新版故事编辑器应该像剧情电话系统的蓝图画布：角色是锚点，CallCard 是节点，出口是触点，触点后面是一组动作；画布负责表达故事结构，浮窗负责编辑细节，引擎负责保证通话结束后的状态推进准确执行。
