# Studio V2.0 UI 需求总览

## 1. 为什么会有 Studio V2.0

AirPC Engine 的核心目标不是做一个普通后台，而是做一套围绕 CallCard 的剧情创作、调试、导出工具。

当前引擎侧已经逐步形成了明确方向：

- 剧情里的每一通电话都由 CallCard 驱动。
- 没有剧情卡时，自由通话也应由 FreeCard / ScheduleCard 等卡片形态承载。
- CallCard 负责入口、目标、上下文、工具能力、出口与状态推进。
- 真正推进状态必须在通话结束后经 WET 与 EffectExecutor 完成。
- 故事包可以理解为章节，章节结束后由 end_story / 章节结束配置决定下一章节的挂卡、时间和角色。

这意味着 UI 不应该只是“能编辑 JSON 文件”或“能调用引擎 API”，而应该成为一个面向故事设计者的 CallCard Studio。

旧 UI 在功能堆叠过程中已经偏离了这个方向。它更像引擎验证台、JSON 管理后台和调试接口集合，而不是一个可长期使用的故事编辑器。因此需要启动 Studio V2.0。

## 2. 旧 UI 的主要缺陷

### 2.1 产品形态错误

旧 UI 的核心问题不是单纯不好看，而是产品定位错位：

- 它把引擎 API 直接暴露给用户。
- 它把 JSON 文件当成主要编辑对象。
- 它把 schema 字段、内部 ID、引擎术语直接放进界面。
- 它没有把“故事创作流程”翻译成清晰的人类工作流。

用户看到的是 packageId、cardId、toolPolicy、condition.op、effect、toClockMs、beginCall、endCall、invokeTool，而不是“故事包、卡片、入口、出口、回拨、状态变化、下一章节”。

### 2.2 故事编辑器不可作为正式创作工具

旧故事编辑器存在明显问题：

- 中央画布被左右栏挤压，无法成为主要工作区。
- 角色泳道只是背景装饰，不是真正的布局模型。
- 卡片拖动、缩放、自动布局后，节点与泳道容易产生错位感。
- Start / End 不是一等可编辑节点，无法承担章节入口和章节结束配置。
- 属性面板仍然大量展示 schema 字段，而不是创作语言。
- JSON 仍是主要对照和兜底编辑方式。
- 卡片、出口、Effect 的创建流程不像故事编辑器，更像结构化 JSON 表单。

### 2.3 调试器不是剧情调试器

旧调试器的问题更严重：

- 左侧控制区混杂了会话启动、时间模拟、Outcome、电话事件、文本对话、工具调用、日志入口。
- 主体区域大量展示 JSON snapshot、effect plan、board、logs。
- 用户无法直观看懂“当前卡片目标是什么、这轮通话完成了什么、命中了哪个出口、执行了哪些 Effect、状态发生了什么变化”。
- 一些调试逻辑仍有 demo 化或硬编码倾向，例如 required beat 与具体 cardId 绑定。

旧调试器更像 JSON 观察器，不像剧情运行器。

### 2.4 ID 生成责任错误

旧 UI 到处要求用户手填 ID，或者用临时字符串拼 ID：

- 创建故事包要求填写 packageId。
- 创建角色要求填写 agentId。
- 创建用户要求填写 userId。
- 新建资产要求填写 assetId。
- 卡片允许直接重命名 cardId。
- Exit / Effect / Fact 有的使用 Date.now().toString(36) 拼 ID。

这不符合创作工具的设计。

用户应该填写标题、显示名、说明、内容。ID 应由系统统一生成，并在高级模式中只读展示。

### 2.5 代码结构已经固化混乱

旧 UI 的代码结构也不适合继续扩展：

- 大页面文件过长，多个页面超过 400 行，调试器页面超过 1000 行。
- 大 Hook 承担过多职责，故事编辑器操作 Hook 接近 1000 行。
- Store 同时保存结构化对象和 JSON 字符串草稿。
- UI 组件中大量 JSON.parse / JSON.stringify。
- 大量类型退化成 unknown / Record<string, unknown>。
- 缺少前端测试、lint、复杂度门禁。
- 视觉 token 和组件规范不足，颜色、布局、按钮形态散落各处。

继续在旧 UI 上修，会把错误架构越修越深。

## 3. 为什么新开项目，而不是在旧方案上改

Studio V2.0 应作为新 UI 项目启动，而不是在旧 UI 上渐进式修改。

原因如下：

### 3.1 旧 UI 的基础抽象不对

旧 UI 的核心抽象是：

- JSON 文件
- API 调用
- 调试 snapshot
- schema 字段

新 UI 的核心抽象应该是：

- 故事包 / 章节
- 角色
- CallCard
- 入口
- 出口
- Effect
- WET 事件
- 状态变化
- 调试时间线

这不是局部重构能解决的问题。

### 3.2 旧 UI 可以保留为引擎验证台

旧 UI 不需要删除。它仍然有价值：

- 可以作为旧功能参考。
- 可以作为引擎 API 验证台。
- 可以作为临时 JSON 管理后台。
- 可以为新 UI 提供接口行为参考。

但它不应该继续承担正式 Studio 的产品目标。

### 3.3 新 UI 需要全新的信息架构

V2.0 需要从“故事设计者如何工作”出发，而不是从“引擎有哪些 API”出发。

这要求重新设计：

- 页面导航
- 编辑器布局
- 调试器流程
- 状态管理
- 保存策略
- 领域模型
- 组件系统
- 测试门禁

因此新开项目更清晰，也更安全。

## 4. 新 UI 的产品定位

Studio V2.0 是 AirPC Engine 的正式创作工具，包含三块核心能力：

### 4.1 CallCard 编辑器

用于创建、编辑、组织故事包中的 CallCard。

它需要支持：

- 创建故事包 / 章节。
- 创建角色。
- 创建剧情卡、自由卡、过场播放卡、延迟外呼卡、定时卡。
- 在画布中拖动、连接、组织 CallCard。
- 以角色锚点和角色归属线展示 CallCard 所属角色。
- 编辑卡片入口、上下文、目标、工具能力、出口。
- 编辑出口条件、出口类型、Effect 列表。
- 编辑章节结束点与下一章节配置。
- 校验故事结构。
- 导出内容包。

### 4.2 CallCard 调试器

用于按真实电话流程运行故事。

它需要支持：

- 选择用户。
- 选择故事包、起始卡、自由通话或当前角色 pending 卡。
- 开始通话。
- 模拟用户对话、未接、占线、提前挂断、播放完成、播放跳过等事件。
- 构建本轮 Outcome。
- 结束通话并触发 WET + EffectExecutor。
- 展示命中的出口。
- 展示 Effect 执行结果。
- 展示角色挂卡 / 卸载卡 / 延迟外呼 / 章节切换等状态变化。
- 支持推进时间，观察延迟外呼和调度结果。
- 高级模式中展示 raw snapshot、WET、effect plan、logs。

### 4.3 内容包管理器

用于管理创作资产和发布流程。

它需要支持：

- 管理故事包。
- 管理角色库。
- 管理音频 / 图片 / 文本资源元数据。
- 管理导入导出。
- 管理校验报告。
- 管理用户测试档案。
- 查看内容包版本与 schema 兼容性。

## 5. 新 UI 的基础技术需求

Studio V2.0 仍以 Web 应用为基础，建议技术选型：

- Next.js 16
- React
- TypeScript
- React Flow / XYFlow
- Zustand 或 Jotai
- Zod
- Monaco Editor 或 CodeMirror，仅用于高级 JSON 查看与源码模式
- Vitest
- Playwright
- ESLint

必须新增：

- 统一 ID 工厂。
- 前端领域模型。
- 编辑器 command 层。
- undo / redo 历史层。
- 保存事务或批量提交策略。
- UI 组件规范。
- 复杂度门禁。

## 6. 新 UI 的关键需求

### 6.1 用户不手填内部 ID

所有内部 ID 必须由系统自动生成：

- storyId / packageId
- cardId
- exitId
- effectId
- agentId
- assetId
- userId
- factId
- scheduleId

用户只填写：

- 标题
- 显示名
- 描述
- 备注
- 剧情内容
- 资源名称

内部 ID 可在高级信息中只读展示。

### 6.2 JSON 不是主编辑方式

V2.0 不允许以 textarea JSON 作为主编辑方式。

JSON 只能用于：

- 高级源码视图。
- 导入导出预览。
- 调试快照。
- 开发者模式。

主编辑必须是结构化表单、画布操作、选择器和可视化配置。

### 6.3 画布是编辑器中心

故事编辑器应采用工作台布局：

```text
顶部：故事名 / 校验状态 / 保存 / 运行调试 / 导出
左侧：固定角色锚点栏
中央：全屏自由 CallCard 画布
底部：轻量工具条
浮窗：卡片模板 / 属性 / 出口动作 / 资源 / 校验 / 小地图 / 调试
```

要求：

- 中央画布必须占主空间。
- 除角色锚点栏外，功能区默认使用绝对定位浮窗。
- 不使用泳道作为默认视图或整理工具。
- 卡片所属角色由顶部角色归属口和角色锚点线表达。
- 拖动卡片不会隐式改变所属角色，必须通过明确的角色归属操作修改。
- 自动布局必须尊重用户手动位置，不能强制贴合泳道。
- Start / End / Chapter End 是一等节点。

### 6.4 CallCard 节点必须人类可读

卡片节点不应只显示 ID 和 schema 字段。

节点应展示：

- 卡片标题
- 所属角色
- 卡片类型
- 入口方式
- 当前目标摘要
- 出口数量
- 是否有播放资源
- 是否有延迟外呼
- 是否有失败兜底

例如：

```text
张老板回拨
角色：张老板
类型：延迟外呼，可提前呼入
目标：确认内存条报价
出口：3
```

### 6.5 属性面板必须按创作语言组织

选中卡片时，属性面板应分组：

- 基本信息
- 入口方式
- 通话目标
- 可明说上下文
- 私有上下文
- 情绪 / 语气
- 工具能力
- 过场播放资源
- 出口列表
- 高级信息

选中出口时，属性面板应分组：

- 出口标题
- 触发条件
- 出口类型
- 优先级
- 动作列表
- 失败 / 兜底策略
- 高级信息

### 6.6 调试器必须是剧情运行器

调试器应按步骤组织：

1. 选择用户。
2. 选择故事包 / 卡片 / 自由通话。
3. 开始通话。
4. 模拟通话过程。
5. 构建 Outcome。
6. 结束通话。
7. 展示出口命中。
8. 展示 Effect 执行。
9. 展示状态变化。
10. 推进时间或进入下一通电话 / 下一章节。

必须突出：

- 当前正在跑哪张 CallCard。
- 本轮通话目标是什么。
- 必达 Beat 完成情况。
- LLM 可用工具有哪些。
- 本轮为什么命中某个出口。
- 本轮对世界状态产生了什么影响。

### 6.7 保存策略必须清晰

V2.0 不应出现“拖一下就写文件、但又有保存按钮”的混乱策略。

建议：

- 编辑器内先修改本地 editor state。
- 用户显式保存。
- 保存前执行校验。
- 保存失败不得污染已保存内容。
- 尽量以 story package 为单位批量保存。
- 保存结果应展示清晰反馈。

### 6.8 校验必须产品化

校验不应只是 error / warning 列表。

应支持：

- 在画布上标记问题节点。
- 在属性面板定位具体字段。
- 区分阻断错误、风险警告、建议优化。
- 提供可读解释和修复建议。
- 支持一键定位到问题卡片 / 出口 / Effect。

### 6.9 高级模式必须隔离

开发者信息必须放到高级模式中：

- raw JSON
- schema 字段名
- internal id
- rendered prompt
- WET raw event
- effect plan JSON
- debug logs

普通创作视图不应默认展示这些内容。

## 7. UI 文案要求

界面文案必须中文优先。

允许在必要时展示原字段名，但格式应为：

```text
故事包 ID（packageId）
卡片 ID（cardId）
工具策略（toolPolicy）
出口条件（condition）
```

禁止主界面直接出现：

- beginCall
- endCall
- invokeTool
- packageId
- cardId
- agentId
- condition.op
- effect
- toolPolicy.mode
- toClockMs
- localNowIso

除非处于高级模式。

## 8. 代码质量要求

Studio V2.0 从第一版开始必须设置工程约束。

### 8.1 文件复杂度

- page.tsx 只做页面组合，不承载完整业务流程。
- page.tsx 超过 120 行告警，200 行为硬上限。
- 普通 UI 组件超过 250 行告警，400 行为硬上限。
- Hook / bis / shell / command 文件超过 250 行告警，400 行为硬上限。
- 单个函数或 Hook 超过 60 行告警，100 行为硬上限。
- 达到告警线应优先按职责拆分；达到硬上限必须拆分。

具体告警线、硬上限和统计口径见 [08-Studio-V2前端工程规则与质量门禁需求.md](./08-Studio-V2前端工程规则与质量门禁需求.md)。达到硬上限必须由 CI 阻断，不能只靠代码审查提醒。

### 8.2 分层要求

建议分层：

```text
domain/      领域模型、schema、纯函数
commands/    编辑器命令
store/       状态管理
services/    API 与文件导入导出
components/  通用 UI 组件
features/    编辑器、调试器、资源管理等功能模块
app/         Next.js 路由入口
```

### 8.3 测试要求

必须覆盖：

- ID 生成。
- 创建卡片。
- 复制卡片。
- 连线生成出口。
- 出口条件编辑。
- Effect 添加 / 删除 / 排序。
- Start / End / Chapter End 行为。
- 保存失败处理。
- 调试器 Outcome 到 Effect 结果展示。

### 8.4 禁止事项

- 禁止在 UI 组件中散落 JSON.parse / JSON.stringify 作为主编辑逻辑。
- 禁止在 UI 组件中手写 ID 生成规则。
- 禁止 Date.now().toString(36) 作为内容 ID。
- 禁止把 API 字段原样暴露为主界面文案。
- 禁止一个页面文件承载多个业务域。
- 禁止调试器以 raw JSON 为主体视图。

### 8.5 注释与自动化门禁

- 公共领域契约、跨层端口、编辑器命令、关键状态转换和非显然副作用必须有意图注释或 JSDoc。
- 特殊字段必须说明单位、时序、所有权、持久化或可空语义。
- 不要求对简单 JSX、局部变量和显然代码机械加注释，也不采用注释密度作为质量指标。
- `eslint-disable`、`@ts-ignore`、`@ts-expect-error` 和门禁豁免必须说明原因；临时项还要写退出条件。
- 后续必须实现 `check:comments`、`check:studio-structure` 和统一 `quality:studio` 命令。
- V2 合并前必须全量通过质量命令；详细脚本契约见 [08](./08-Studio-V2前端工程规则与质量门禁需求.md)。

## 9. 需要避免的问题

Studio V2.0 必须避免重复旧 UI 的问题：

- 不要做成 JSON 编辑器。
- 不要做成 API 调用控制台。
- 不要让用户手填内部 ID。
- 不要让画布被侧栏挤压到不可用。
- 不要引入泳道作为默认布局。
- 不要把 Start / End 做成不可编辑摆设。
- 不要中英文和 schema 字段混杂。
- 不要把调试器做成 snapshot 展示器。
- 不要把保存、自动保存、刷新、重拉混在一起。
- 不要没有测试就继续堆功能。
- 不要让老 UI 的 store / page / BIS 架构迁移进 V2.0。

## 10. 与引擎改进需求包的关系

当前目录下的引擎小改和增强计划，与 Studio V2.0 UI 需求共同组成新的需求包。

二者关系：

- 引擎负责提供稳定能力、语义和接口。
- Studio V2.0 负责把这些能力翻译成可创作、可调试、可发布的产品体验。
- UI 不应弥补引擎语义缺口。
- 引擎也不应要求用户通过 UI 手工维护内部状态。

Studio V2.0 的目标不是把引擎细节全部暴露出来，而是让故事设计者能直观管理 CallCard 故事链条。

## 11. 第一版验收标准

Studio V2.0 第一版至少应满足：

- 可以创建故事包，ID 自动生成。
- 可以创建角色，ID 自动生成。
- 可以创建 CallCard，ID 自动生成。
- 可以在画布中通过角色锚点和归属线组织卡片。
- 可以连接卡片并生成出口。
- 可以编辑卡片上下文、目标、入口方式、工具策略。
- 可以编辑出口条件与 Effect。
- 可以配置 End / Chapter End。
- 可以保存并校验故事包。
- 可以进入调试器跑一轮通话。
- 可以看到出口命中、Effect 执行结果和状态变化。
- JSON 只作为高级视图存在。

达到这些要求后，Studio V2.0 才能算是 CallCard 编辑器，而不是旧 UI 的换皮版本。
