# CallCard 调试器界面设计导向稿

> 版本定位：本文定义 Studio V2.0 调试器的设计方向。  
> 调试器不是 JSON 展示器，也不是 API 控制台，而是剧情电话流程的可视化运行器。

## 1. 调试器要解决什么

CallCard 调试器要让故事设计者看懂一轮电话如何推进剧情：

1. 当前是自由通话还是剧情通话？
2. 这通电话由哪张 CallCard 驱动？
3. 用户或角色做了什么？
4. WET 产出了什么结果？
5. 命中了哪个出口？
6. EffectExecutor 执行了哪些动作？
7. 哪些角色被挂卡、卸载卡、安排延迟外呼？
8. 故事是否结束，是否进入下一章节安排？

这套问题应该被界面直接回答，而不是让用户在 raw JSON 里找。

## 2. 调试器基础形态

调试器可以有两种入口：

- 从故事编辑器中打开调试浮窗。
- 从首页或故事包列表进入独立调试器页面。

第一版建议先做独立调试器页面，同时支持从编辑器带入当前故事包和当前卡片。

页面布局建议：

```text
顶部：故事包 / 用户档案 / 当前模式 / 重置 / 返回编辑器
左侧：调试场景设置
中部：通话运行区
右侧：角色状态与挂卡队列
底部：WET + Effect 执行时间线
高级抽屉：raw snapshot / logs / JSON
```

## 3. 调试场景设置

调试开始前，用户需要选择一个场景。

可选场景：

- 用户呼入某角色。
- 某角色主动外呼用户。
- 延迟外呼触发。
- 过场播放卡触发。
- 自由通话 fallback。

场景设置字段：

- 用户档案。
- 目标角色。
- 起始 CallCard，可自动从目标角色 pending 卡推断。
- 是否使用当前角色挂卡状态。
- 是否重置故事状态。

不要求用户手填：

- sessionId。
- pending instance id。
- packageId。
- WET raw payload。

## 4. 通话运行区

通话运行区是调试器的主区域。

需要展示：

- 当前通话角色。
- 当前 CallCard。
- 通话类型：自由通话 / 剧情通话 / 过场播放 / 延迟外呼。
- 本轮目标。
- 当前上下文摘要。
- 用户输入或模拟事件。
- 角色回复摘要。
- 通话结束按钮。

第一版不需要做真实语音通话，可以使用文本模拟和事件按钮。

事件按钮建议：

- 用户正常沟通。
- 用户拒绝。
- 用户无应答。
- 用户提前挂断。
- 角色完成目标。
- 播放完成。
- 播放跳过。
- 工具调用成功。
- 工具调用失败。

这些按钮应该是剧情语义按钮，不是底层 API 按钮。

## 5. 出口命中展示

通话结束后，调试器必须清楚展示：

- LLM / WET 判定结果。
- 命中的出口。
- 未命中时使用的兜底出口。
- 出口触发理由。
- 出口动作列表。

出口命中不应只展示 exitId。

示例：

```text
命中出口：聊崩了
原因：用户拒绝继续沟通，并表达不信任
动作：
1. 给小雨挂卡「安抚用户」
2. 给张老板挂卡「解释失败」
3. 设置世界状态「澜星关系紧张」
```

## 6. Effect 执行结果

EffectExecutor 执行必须被调试器可视化。

每个 effect 显示：

- 动作类型。
- 目标对象。
- 输入摘要。
- 执行状态。
- 是否 critical。
- 成功或失败原因。

状态：

- 等待执行。
- 执行中。
- 成功。
- 失败但可继续。
- critical 失败并中断。

调试器必须表达「不是 fire-and-forget」的语义：Effect 需要执行完成后才能认为这一轮状态推进完成。

## 7. 角色状态区

右侧角色状态区展示调试后的世界状态。

每个角色展示：

- 当前是否可自由通话。
- 是否有 pending CallCard。
- 是否有待外呼。
- 是否有延迟外呼计时。
- 当前挂卡标题。
- 当前挂卡来源。

点击角色可展开：

- pending 卡列表。
- 最近变化。
- 相关出口。

不要把角色状态做成原始 store dump。

## 8. 时间推进

剧情里通常没有循环定时外呼，但会有延迟外呼。

调试器第一版应支持：

- 查看延迟外呼队列。
- 手动推进时间。
- 触发到期外呼。
- 用户提前呼入目标角色时，验证是否消费对应外呼卡。

这个能力非常重要，因为延迟外呼卡可以被用户提前呼入自然承接。

界面表达：

```text
张老板将在 2 分钟后外呼
用户现在呼入张老板：将使用「张老板回拨」卡
```

## 9. 与编辑器联动

从编辑器进入调试器时：

- 自动带入当前故事包。
- 如果选中了卡片，默认从该卡开始。
- 调试结果可以回写为调试标记，但不应自动改剧情结构。
- 用户可以点击调试中的卡片回到编辑器定位。

从调试器回编辑器时：

- 高亮本次经过的卡片。
- 高亮命中的出口。
- 标记失败 effect 或警告点。

## 10. 高级视图

高级视图可以提供：

- WET raw result。
- Effect plan。
- Runtime snapshot。
- Logs。
- JSON。

但高级视图必须在抽屉或弹窗中，不能作为主界面。

使用 Monaco / CodeMirror 格式化展示，不使用 textarea。

## 11. 调试记录

每次调试完成后，应形成一条调试记录。

记录包含：

- 故事包。
- 起始卡。
- 目标角色。
- 通话类型。
- 命中出口。
- effect 执行摘要。
- 角色状态变化摘要。
- 时间。

调试记录用于复盘，不是日志全文。

## 12. 第一版不做的事情

第一版暂不要求：

- 时间轴回放。
- 真实语音通话录音。
- 完整 LLM 对话仿真。
- 多用户并发调试。
- 性能分析。
- 调试脚本录制。

这些可以后续单独设计。

## 13. 验收标准

第一版至少满足：

- 可以选择故事包和目标角色开始调试。
- 可以模拟用户呼入或角色外呼。
- 可以看到当前 CallCard 和本轮目标。
- 可以结束通话并触发 WET。
- 可以看到命中的出口。
- 可以看到出口动作列表。
- 可以看到 EffectExecutor 执行结果。
- 可以看到角色挂卡 / 卸载卡 / 延迟外呼变化。
- 可以手动推进时间并触发延迟外呼。
- 高级 JSON 不作为主界面。

## 14. 避免事项

- 不要把调试器做成 JSON snapshot 页面。
- 不要把所有按钮写成 API 名。
- 不要把 WET 和 EffectExecutor 隐藏在日志里。
- 不要只展示成功，不展示失败原因。
- 不要让用户手工构造 raw outcome。
- 不要忽略延迟外呼和提前呼入的测试场景。

## 15. 一句话总结

调试器要像剧情电话的运行回放器：它展示一轮通话如何命中出口、执行动作、改变角色状态，而不是展示一堆引擎内部 JSON。
