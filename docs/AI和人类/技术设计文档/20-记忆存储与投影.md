# 20. 记忆存储与投影（SQLite）

相关：[12](../需求/12-记忆模型.md) · [13](../需求/13-工具与能力注册.md) · [05](./05-技术选型与工程边界.md) · [18](./18-部署拓扑与BS架构.md)

> **结论：** Content = JSON 文件；**Memory = SQLite**（经 `MemoryPort`）。  
> 存得下 ≠ 查得起、注得起。表结构与预算可随实现微调；本文为 v1 初稿。

## 1. 职责切分

| 数据 | 存储 | 说明 |
|------|------|------|
| 故事包 / 角色 Def / 卡 / 资产 meta | JSON 文件 | 作者向、可 diff |
| Profile 薄存档（Board、story 变量、research、schedule…） | JSON `profile.save.json` | **不含**万级 episodic 数组 |
| NPC 记忆条目 + rollup | **SQLite** | 按 userId+agentId 索引查询 |

路径建议（Studio / 本机）：

```text
data/
  users/<userId>/profile.save.json
  memory/
    memory.sqlite          # 或多文件 users/<userId>/memory.sqlite
```

壳侧可换路径；引擎只认 MemoryPort。

## 2. 表结构初稿

```sql
-- 明细（episodic / semantic 等可用 layer 区分，或分表；v1 可用一张 entries）
CREATE TABLE memory_entries (
  id            TEXT PRIMARY KEY,
  user_id       TEXT NOT NULL,
  agent_id      TEXT NOT NULL,
  layer         TEXT NOT NULL,  -- episodic | semantic | commitment | …
  kind          TEXT,           -- call_summary | vignette | beat | …
  text          TEXT NOT NULL,
  at            TEXT NOT NULL,  -- 事件/通话时刻 ISO
  created_at    TEXT NOT NULL,
  updated_at    TEXT NOT NULL,
  call_id       TEXT,
  importance    REAL,
  fact_id       TEXT,
  expires_at    TEXT,           -- affect 等
  status        TEXT,           -- commitment
  payload_json  TEXT            -- 其余层字段
);

CREATE INDEX idx_mem_user_agent_at
  ON memory_entries(user_id, agent_id, at DESC);

CREATE INDEX idx_mem_user_agent_layer_at
  ON memory_entries(user_id, agent_id, layer, at DESC);

-- 温层摘要（老记忆总汇）
CREATE TABLE memory_rollups (
  id            TEXT PRIMARY KEY,
  user_id       TEXT NOT NULL,
  agent_id      TEXT NOT NULL,
  period_kind   TEXT NOT NULL,  -- month | quarter
  period_key    TEXT NOT NULL,  -- 2026-07 | 2026-Q3
  range_from    TEXT NOT NULL,
  range_to      TEXT NOT NULL,
  summary       TEXT NOT NULL,
  created_at    TEXT NOT NULL,
  updated_at    TEXT NOT NULL,
  UNIQUE(user_id, agent_id, period_kind, period_key)
);

CREATE INDEX idx_rollup_user_agent
  ON memory_rollups(user_id, agent_id, range_to DESC);

-- 冷层召回：FTS5（生产路径）；写入 entries / 可选 rollup 时同步维护
CREATE VIRTUAL TABLE memory_entries_fts USING fts5(
  entry_id UNINDEXED,
  user_id UNINDEXED,
  agent_id UNINDEXED,
  text,
  kind UNINDEXED,
  at UNINDEXED,
  tokenize = 'unicode61'   -- 中文可后续换更好分词；v1 先可用
);

-- 关系 / profileNotes 等低频块可用小表或 entries.layer
```

`relational` / `affect` / `profileNotes`：v1 可用 `memory_entries.layer` + `payload_json`，或另表；以可按 user+agent 读取为准。

**FTS 同步：** `INSERT/UPDATE/DELETE memory_entries` 时同步 FTS；rollup 摘要可选同步（`kind='rollup'`），便于粗粒度命中。单测/极简环境可降级 `LIKE`，生产以 FTS 为准。

## 3. 每通带什么（程序）

```text
beginCall
  → MemoryPort.projectForCall({ userId, agentId, nowIso })
       SQL：最近 N 条 call_summary + 最近 M 条 vignette
            + 未过期 affect + relational/profileNotes
            + 最近 1～2 条 rollup
       → 拼 softText（含相对时间）≤ 预算
  → Composer 叠进 RenderedPrompt.softContext
  → 禁止全表 SELECT *
```

建议默认：`N=5`，`M=8`，热层 ≤ ~2000 字（见需求 12 §2.3）。

## 4. 老记忆何时总汇

```text
MemoryCommit 写入本通 episodic 成功
  → rollupIfNeeded(endedAt)
       若本通跨入新 month/quarter，或该 period 尚无 rollup / 过期
       → 仅聚合该 period 内明细（有界 SQL），UPSERT memory_rollups
后台（可选）
  → 补算缺失 period；低优先级，不挡通话
```

**禁止：** 接通瞬间对十年明细做一次 LLM 大总结。

## 5. 冷层召回算法（`search_memory` / `get_memory_by_id`）

产品契约见 [13 §7.1b](../需求/13-工具与能力注册.md)；端口类型见 [12](../需求/12-记忆模型.md)。  
**与 rollup 无关：** rollup 是挂机后增量记账；本节省通话中 FC 查库。

### 5.1 目标与边界

| 做 | 不做 |
|----|------|
| 按需查 **本 user × 本 agent** 的 SQLite | 十年明细灌进本通 prompt |
| 关键词 ± 时间窗，有界结果 | **向量语义检索**（v1 后置） |
| 短列表 → 可选按 id 取全文 | 通话中写库 / 登记 Exit |
| 超时与空结果可恢复 | beginCall 热路径全表扫 |

### 5.2 端到端时序

```text
用户：「还记得两年前你说过 xxx 吗？」
  → 热层无此条 → 模型 FC search_memory
  → ToolRuntime：behavior=session_local，不进 Exit
  → MemoryPort.search（userId/agentId 由 Host 注入，忽略模型伪造）
  → SQLite FTS + 过滤 + LIMIT
  → 工具结果：≤10 条短 Hit（摘要 ≤200 字）
  → 可选 FC get_memory_by_id → 单条正文（≤500 字）
  → 模型用工具结果接话
```

失败：工具错误字符串回模型；**不**拖死通话、**不**崩 Session。

### 5.3 入参校验

```text
search：text_query 与 (from|to) 至少一组有效
  max_results 默认 5，硬夹紧到 [1, 10]
  非法 ISO / 双无 → 拒查，不碰库
getById：必须同 user+agent 命中；否则未找到
```

`userId` / `agentId` **仅** Session 注入；v1 **禁止**跨 agent 搜。

### 5.4 `search` 步骤

```text
1. 候选召回（内部宽限，不回模型）
   有 text_query:
     FTS MATCH（规范化+转义）AND user+agent
     [AND at ∈ [from,to]] [AND kind IN kinds]
     内部上限：maxR*5 或 50
   仅时间窗:
     SQL WHERE user+agent + at 窗 + kinds
     ORDER BY at DESC LIMIT maxR*5
     （可含 rollup 行，若已同步 FTS/表）

2. 打分排序
   有 FTS: 主 = bm25/rank；次 = at 距窗中心或 now 更近
   仅时间窗: at DESC
   可选 kind 权重: call_summary ≥ vignette ≥ semantic ≥ rollup ≥ beat

3. 截断投影
   取前 maxR；Hit.text 截断 ≤200 字（逼需要全文时走 getById）

4. 空结果 → []；包装层可附 hint「放宽时间或换关键词」

5. 查询预算建议 ≤200～500ms；超时 → 工具错误
```

**query 规范化：** trim；过短（如 1 字）可拒或强制要时间窗；去无意义引号；转义 FTS 特殊字符。中文 v1 可整句 MATCH，效果差再加分词/片段 OR。

**降级：** 无 FTS 时用 `LIKE '%'||q||'%'` + 同上过滤 + LIMIT；生产路径以 FTS 为准。

### 5.5 与热层 / rollup

| 路径 | 何时 | 读什么 |
|------|------|--------|
| `projectForCall` | beginCall | 最近 N/M + 近 1～2 rollup，有字符预算 |
| `search_memory` | 通话中 FC | 条件查 entries（及可选 rollup），LIMIT≤10 |
| `rollupIfNeeded` | Commit 后 | 只更新当前月/季桶 |

`search` **不依赖**先有 rollup；可直接命中很老的 `call_summary`。

### 5.6 审计

`data/logs/*.jsonl`：工具名、参数摘要、hit 数、耗时；可记 id，不强制记全文。

## 6. 迁移与演进

- SQLite → Postgres：换 MemoryPort 实现，表语义保持（FTS → 对应全文索引）  
- **不做**「先把 episodic 堆进 JSON Profile 再迁库」  
- 导出调试包：可导出 sqlite 切片或按时间窗导出 JSONL  
- **后置：** embedding / 向量召回（字面 FTS 不够时再开）

## 7. 验收摘录

- [x] episodic 不在 `profile.save.json` 大数组里  
- [x] `projectForCall` 有字符/条数预算；单测禁止全表注入  
- [ ] Commit 后可增量 rollup（只碰相关 period）  
- [x] search：无关键词且无时间窗 → 拒查；`max_results` 硬 ≤10  
- [x] 热层未注入的旧条，仅靠 FC + FTS（或文档化 LIKE 降级）可查到  
- [x] 跨 user / 跨 agent → 0 命中  
- [x] getById 正文 ≤500 字；search Hit 摘要 ≤200 字  
- [x] 单测禁止 search 实现 `SELECT` 全表无 LIMIT
