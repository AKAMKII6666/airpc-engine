# Studio V2.0 · 第一步到第二步之间的休整计划

> 状态：已收口（REST-G/Q/E/P/A 全部 ✅）。  
> 性质：第一步后的质量休整与缺口修复，不属于第二步业务开发。  
> 原则：本计划全部收口前，不开始第二步的真实保存、Host 对接和调试闭环。

## 1. 为什么需要休整

第一步已经完成了 ScheduleCard、延迟外呼、章节切换、EffectSink、Studio V2 骨架和前端质量门禁的主体实现，并有自动化测试支撑。

但独立复核发现，当前仍存在两类问题：

1. **明确的功能/自动化缺陷**：`general-batch-exe` 终审协议冲突；recurring 卡引用只检查“有 ID”而没有在运行时保证可解析；ScheduleCard 开始通话时误写 StorySave。
2. **工程治理缺口**：Studio V2 已有结构门禁，但引擎没有等价门禁。第一步的引擎任务只以 `npm test` 为出口，导致功能和测试完成了，核心文件却继续增长。

本计划的目标不是否定第一步，而是在进入第二步前补齐“功能正确、结构可持续、自动化能真正收口”三条链路。

## 2. 本计划必须解决的问题

### 2.1 自动化编排

- `[P1]` `general-batch-exe` 终审提示要求通用 `batchId: "<n>"`，编排器却要求 `batchId: "final"`，成功终审因此进入 `BLOCKED`。
- skill 的 mock 测试写死了正确值，没有覆盖真实 Prompt 对 Agent 的约束，导致自测通过而真实运行失败。

### 2.2 引擎语义

- `[P1]` recurring 只验证存在 `scheduleCardId` 或 `cardId+packageId` 字符串，没有在动态创建、旧 Profile 恢复和 tick 物化时统一保证引用存在、卡类型正确、角色归属正确。
- `[P2]` ScheduleCard `beginCall` 会进入 `activateStoryOnBegin`，在 `profile.stories.__schedule__` 下写入 StorySave；这与“ScheduleCard 是非剧情 FreeCard 超集”冲突。

### 2.3 引擎可维护性

当前关键文件规模已经很高：

| 文件 | 当前约数 | 风险 |
|---|---:|---|
| `packages/rpg-engine/src/host/createEngineHost.ts` | 1465 行 | Host 装配、Resolver、Session、调试和持久化继续耦合 |
| `packages/rpg-engine/src/runtime/scheduleTick.ts` | 785 行 | once、recurring、时钟、章节清场混在同一模块 |
| `packages/rpg-engine/src/runtime/effectExecutor.ts` | 706 行 | Effect 分派、章节切换、调度和媒介写口集中 |

现有规则能限制引擎依赖 React、测试位置和目录平铺，但没有限制单文件、单函数、圈复杂度和超大遗留文件继续增长。第一步 E 批 verify 全部为 `npm test`，Reviewer 也主要检查行为和测试，没有结构出口。

## 4. 总体解决方案

执行顺序固定为：

```text
修 gbx 终审协议
  → 建 engine 规则和结构门禁
  → 记录历史基线并启用止血
  → 在新门禁下修 recurring
  → 在新门禁下修 ScheduleCard StorySave
```

## 5. 方案 A：修复 general-batch-exe 终审协议

### 5.1 根因

- `final-reviewer` 复用了通用 Review JSON 模板，模板中的 `batchId` 是 `"<n>"`。
- Prompt 同时展示 `Current batch number: 8`，Agent 自然填写 `"8"`。
- Orchestrator 的正确契约是 `role=final-reviewer + batchId=final + 当前 reviewRunId`。
- mock runner 直接生成 `batchId=final`，没有经过真实 Prompt 解释，因此测试没有暴露矛盾。

### 5.2 修复要求

- `buildPrompt(final-reviewer)` 必须明确写出：`batchId` **只能**是字符串 `"final"`。
- Batch Reviewer 和 Final Reviewer 使用不同的 JSON 契约模板，不再依赖含糊占位符。
- Prompt 中不要向 Final Reviewer展示容易误用的“当前批次号”，或明确标注该数字不可写入 `batchId`。
- `validateReviewReport` 继续严格校验，不降低为兼容 `"8"`；问题应修在生产者契约，不应放松消费者。
- 增加真实 Prompt 契约测试：构造 final-reviewer Prompt，断言其中只有 `batchId: "final"` 的要求，不出现 `batchId: "<n>"`。
- 增加一次接近真实 Agent 输出的测试：报告写 `batchId: "8"` 必须失败，写 `"final"` 必须进入 `READY_FOR_MANUAL_QA`。
- 修复后从当前 `BLOCKED` 状态按 skill 既有恢复协议继续，不手改 `STATE.json` 冒充成功。

### 5.3 目标落点

- `.cursor/skills/general-batch-exe/lib/prompts.js`
- `.cursor/skills/general-batch-exe/lib/orchestrator.js`（仅在契约封装确有必要时）
- `.cursor/skills/general-batch-exe/docs/02-review-json.md`
- `.cursor/skills/general-batch-exe/tests/`

### 5.4 验收

```text
cd .cursor/skills/general-batch-exe && npm test
gbx 从当前 BLOCKED 恢复后可通过 FULL_REVIEW 到 READY_FOR_MANUAL_QA
STATE.blockedReason 不再出现 review report batchId must be final
```

## 6. 方案 B：recurring 引用必须真正可解析

### 6.1 正确语义

“有卡引用”必须表示：

```text
引用字段完整
  + 目标卡真实存在
  + 卡类型允许用于 recurring
  + 卡归属角色与任务角色一致
  = 可物化的 recurring
```

仅有一个非空 ID 不算可解析。

### 6.2 建立统一解析口

引擎应建立单一的 Schedule/Free 卡引用解析能力，供以下路径共同使用：

- 内容包校验 `validatePackage`。
- 动态 FC / Effect 创建 recurring intent。
- Profile 载入后的旧 intent 检查。
- `materializeRecurringOccurrences` / clock tick。
- `resolve(agent_outbound)` 前的最后防线。

建议形成纯解析器或端口：

```text
resolveScheduledCardReference({ agentId, scheduleCardId | packageId+cardId })
  → { ok: true, packageId, cardId, cardKind, ownerAgentId }
  → { ok: false, code, reason }
```

EffectExecutor 不应自行读取磁盘。由 Host/Workspace 提供只读解析端口，Executor 与 scheduler 消费统一结果，避免在多个模块复制判断。

### 6.3 创建阶段

- 动态 `schedule_recurring_call` 在写入 Profile 前必须解析目标卡。
- 未知卡、错误 `cardKind`、错误 package sentinel、owner 不一致时，本 effect 失败，不写 recurring intent。
- `scheduleCardId` 只能解析到 `characters/schedule-cards` 中 `cardKind=schedule` 的卡。
- 显式 Free fallback 只能解析到允许的 FreeCard；不得指向 StoryCard。
- 不允许通过 `topicHint` 或未来 resolve 兜底掩盖无效引用。

### 6.4 Workspace 与恢复阶段

- 加载 `characters/schedule-cards/**` 时必须强制 `cardKind=schedule`。
- 加载 FreeCard 时也应验证其允许类型，避免目录名和实际 cardKind 分叉。
- Profile 中已有的 recurring intent 在恢复或 tick 前重新解析。
- 对旧/非法 intent：标记 `disabled` 并记录稳定错误原因；不得生成 once、pending 或 fired 结果。
- 卡在任务创建后被删除/改类型时，下一次 tick 必须安全 disabled，不能先通知壳外呼再在 resolve 阶段失败。

### 6.5 目标落点

建议新增独立解析模块，避免继续扩大 `scheduleTick.ts` 和 `effectExecutor.ts`：

- `packages/rpg-engine/src/schedule/scheduleCardReferenceResolver.ts`（或等价明确领域目录）
- `packages/rpg-engine/src/workspace/`：只负责从 Workspace 查询卡
- `packages/rpg-engine/src/runtime/effectExecutor.ts`：仅调用注入解析口，不实现查找规则
- `packages/rpg-engine/src/runtime/scheduleTick.ts`：仅消费已验证引用/解析结果
- `packages/rpg-engine/tests/schedule/` 与必要 Host E2E

最终落点可按现有目录调整，但执行索引必须先写清目标模块和禁止继续堆入的模块。

### 6.6 必测场景

- 未知 `scheduleCardId`：创建失败，不写 intent。
- schedule 目录中 `cardKind=story/free`：加载或解析失败。
- `ownerAgentId` 与 recurring `agentId` 不一致：创建失败。
- 显式 `packageId+cardId` 指向 StoryCard：失败。
- 合法 ScheduleCard：创建、tick、pending、resolve、beginCall 全链通过。
- intent 创建后卡文件被删除：下一 tick 标 disabled，无 fired、无 pending。
- 旧 Profile 中只有 topicHint 或无效 ID：disabled，无可观测 outbound。
- 双 userId 隔离仍然通过。

## 7. 方案 C：ScheduleCard 不得写 StorySave

### 7.1 正确分类

通话是否属于剧情，不能只通过 `packageId !== "__free__"` 判断。

建议建立统一分类函数：

```text
isNarrativeCall(result/session)
  StoryCard / story_pending / simulate story → true
  FreeCard / ScheduleCard / __free__ / __schedule__ → false
```

Host 的 begin、end、StorySave、ActiveStoryLock 和 FreePostPipeline 都应使用同一分类口径，避免 begin 和 end 各写一套条件。

### 7.2 修复要求

- ScheduleCard `beginCall` 不调用 `activateStoryOnBegin`。
- 不创建或更新 `profile.stories.__schedule__`。
- 不获取、继承或释放剧情 ActiveStoryLock。
- 挂机仍进入 FreeCallPostPipeline。
- ScheduleCard 的 pending/once 消费仍按调度生命周期执行。
- StoryCard、章节 next 和延迟剧情外呼的 StorySave 激活行为保持不变。

### 7.3 目标落点

- 新增小型纯分类模块，例如 `packages/rpg-engine/src/runtime/classifyCall.ts`。
- `createEngineHost.ts` 只调用分类结果，不继续增加多处分支表达式。
- 对应测试放 `packages/rpg-engine/tests/host/` 或 `tests/runtime/`。

### 7.4 必测场景

- ScheduleCard begin 前后都不存在 `profile.stories.__schedule__`。
- ScheduleCard 不产生 ActiveStoryLock。
- ScheduleCard endCall 仍走 FreePostPipeline。
- 连续多次 recurring 不积累 Schedule StorySave。
- StoryCard begin 仍正常激活 StorySave。
- `end_story.next` 入口真正 begin 时仍激活下一章节。

## 9. 方案 E：check:engine-structure 与 quality:engine

### 9.1 结构门禁范围

新增：

```text
scripts/engine-quality/check-engine-structure.mjs
scripts/engine-quality/engine-quality-baseline.json
scripts/engine-quality/tests/**
```

`check:engine-structure` 至少检查：

- 手写 TS 文件有效行数。
- 单函数有效行数。
- 圈复杂度。
- `src/**` 是否平放测试。
- 引擎是否导入 React/Next/DOM（可复用现有 import 门禁结果）。
- Host 是否直接新增可识别的领域职责依赖。
- 目录职责聚类与 barrel 边界。
- 无理由 `eslint-disable` / TypeScript suppress / 质量豁免。
- 历史基线文件是否出现指标净增长。

脚本必须输出：

```text
规则编号  文件:行:列  当前值  允许值/基线值  修复建议
```

### 9.2 新代码阈值

第一版阈值：

| 检查对象 | 告警线 | 硬上限 |
|---|---:|---:|
| 普通引擎源码文件 | 350 行 | 500 行 |
| Host / composition 文件 | 400 行 | 600 行 |
| 单个函数 | 60 行 | 100 行 |
| 圈复杂度 | 10 | 15 |
| 测试文件 | 450 行 | 700 行 |

这些阈值约束新文件和未进入历史基线的文件。不能为了迁就现有超大文件把全局硬上限提高到 1500 行。

### 9.3 历史基线止血

对于当前已经超过硬上限的文件：

- 首次落地脚本时记录有效行数、最大函数行数、最大圈复杂度。
- 基线只记录明确文件，不允许目录通配。
- 不要求本休整期一次拆完。
- 文件未触碰时允许保持现状。
- 文件被任务触碰后，任何受控指标不得高于基线。
- 新增功能导致文件有效行数、最大函数或复杂度净增长，门禁失败。
- 修改已超限函数时，至少不得增长；能安全下降时应下降。
- 从基线移出的文件不得重新加入。
- 每条基线包含原因、记录日期、责任域和退出条件。

这就是当前阶段的“止血”：不承诺立即偿还全部历史债，但从门禁上线开始不允许债务继续增加。

### 9.4 quality:engine 统一入口

根 `package.json` 最终提供：

```text
npm run check:engine-structure
npm run quality:engine
```

`quality:engine` 必须汇总：

```text
check:engine-imports
引擎 typecheck
check:engine-structure
引擎 Vitest
结构门禁脚本自测
```

如果后续建立引擎 ESLint，加入同一入口；本计划不得用“以后再加 lint”推迟上述五项。

任一硬门禁失败，`quality:engine` 返回非零退出码。

### 9.5 门禁脚本自身测试

至少覆盖：

- 新文件超过文件硬上限失败。
- 新函数超过函数硬上限失败。
- 圈复杂度超过硬上限失败。
- 历史超限文件保持基线通过。
- 历史文件增加一行或复杂度上升失败。
- 历史文件下降通过，并可更新/移除基线。
- 大目录 allowlist 被拒绝。
- 生成文件和构建产物正确排除。
- 错误位置可被 `general-batch-exe` 的 verify location parser 识别。

## 10. 方案 F：高风险任务必须声明目标模块

以后涉及以下领域的任务视为高风险：

- Resolver / pending / ActiveStoryLock。
- EffectExecutor / WET / EffectSink / ledger。
- Schedule once / recurring / clock。
- StorySave / end_story / chapter next。
- Profile 持久化与 userId 隔离。
- schemaVersion / 内容包加载。

执行索引中的高风险任务必须同时写明：

```text
行为目标：要实现什么
目标模块：主要逻辑放在哪里
允许修改：哪些文件/目录
禁止落点：不得继续塞入哪些上帝模块
状态不变量：必须保持什么
回归测试：至少哪些负向/恢复场景
verify：npm run quality:engine
结构预算：是否触碰历史基线，指标只能持平或下降
```

缺少上述信息时，`general-batch-exe` dry-run 或 Reviewer 应将任务视为不可执行，而不是让 Executor 自行猜落点。

## 11. 方案 G：Reviewer 增加结构审查出口

引擎批 Reviewer 不能只检查“测试是否通过”。报告必须增加：

```text
behaviorChecks     行为契约是否完成
negativeChecks     非法输入/恢复路径是否覆盖
structureChecks    模块落点是否符合任务声明
baselineDelta      历史基线指标是否增长
qualityEngine      quality:engine 是否通过
```

以下情况至少记为 major，并进入 Fix：

- 高风险逻辑落入任务声明之外的模块。
- 触碰超大历史文件并产生净增长。
- 新增函数超过硬上限。
- 只增加 happy path，没有非法引用、恢复态或幂等测试。
- `npm test` 通过但 `quality:engine` 未运行或失败。
- 用 allowlist、disable、压行或删注释通过门禁。

Final Reviewer 还必须抽查：

- 所有引擎批的 `quality:engine` 历史报告。
- 基线 delta 是否为 0 或下降。
- 是否存在“任务已勾选但 verify 失败”的残留状态。
- Review JSON 的 role、batchId、reviewRunId 是否符合当前阶段契约。

## 12. 引擎批 verify 新口径

从本计划开始：

```text
引擎代码任务 verify = npm run quality:engine
```

不再使用单独的 `npm test` 作为引擎任务完成出口。

允许更窄命令用于开发中的快速反馈，但任务勾选前必须执行完整 `quality:engine`。Batch Verify 和 Full Verify 都必须使用完整入口。
