# 16. 错误处理、幂等与事件顺序

相关：[02](./02-运行时协议.md) · [01](./01-CallCard数据模型.md) · [15](./15-动态出口与通话中工具事件.md) · [19](../技术设计文档/19-引擎宿主与会话模型.md)

## 1. Effect 计划执行策略（v1）

一组出口的 `effects[]` **不是**数据库事务回滚模型。

**默认策略：逐条执行；失败记录并继续（best-effort）。**

```ts
interface EffectPlanResult {
  exitId: string
  status: "completed" | "completed_with_errors" | "aborted"
  results: Array<{
    effectId: string
    ok: boolean
    skipped?: boolean
    error?: string
  }>
}
```

| 规则 | 说明 |
|------|------|
| 顺序 | 按数组顺序执行 |
| 单条失败 | 写 `story.effect.failed`，**继续**下一条（默认） |
| `critical: true`（可选字段） | 该条失败则 `status=aborted`，**停止**后续 |
| 幂等 | 已成功 key 再跑 → `skipped`；权威账本在 **CallSession.effectLedger**，成功后摘要可固化到 Profile（见 [19 §9](../技术设计文档/19-引擎宿主与会话模型.md)） |
| 回滚 | v1 **不做**自动回滚；调试器展示失败项；排障看 `data/logs/*.jsonl` |

```ts
interface EffectBase {
  id: string
  effect: string
  critical?: boolean
}
```

## 2. 事件顺序（钉死）

```text
story.card.activated
  → （通话中）exit_candidate.registered …
  → story.card.evaluated          // Outcome 已生成
  → story.card.exit_selected      // ExitSelector 已选定
  → story.effect.executed | skipped | failed  // 逐条
  → story.card.completed          // effect plan 终态之后（含 completed_with_errors）
  → story.card.exited             // 摘要：exitId + plan.status（可与 completed 合并实现，但语义上 completed 在 plan 后）
```

| 事件 | 含义 |
|------|------|
| `evaluated` | 有结构化 Outcome，尚未定出口 |
| `exit_selected` | 已选 exitId / 动态候选 |
| `effect.*` | 单条副作用 |
| `completed` | **本轮卡推进结束**（plan 跑完或 aborted），不是「刚挂机」 |

挂机瞬间若尚未评估，可另有壳事件 `call.ended`；不要把 `call.ended` 等同于 `story.card.completed`。

## 3. 与幂等 key

```text
{storyInstanceId}:{cardInstanceId|cardId}:{exitId}:{effect.id}
```

plan 部分失败后重试：仅未成功的 effect 应可再执行；已成功保持 skipped。

## 4. 调试器展示

- Effect plan 面板：每条 ok/skipped/failed
- `effectPlan.status` 标色
- 禁止静默吞掉失败

## 5. 非目标

- 分布式两阶段提交
- 失败自动补偿反向 effect（作者应用 recovery 出口表达）
