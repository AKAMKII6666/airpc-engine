# 12. 记忆模型

相关：[10-用户与存档](./10-用户与存档.md) · [01-CallCard数据模型](./01-CallCard数据模型.md) · [50-对话与模型适配](./50-对话与模型适配.md)

## 1. 原则

- 记忆是 **per-user、per-agent**（逻辑归属仍挂在该用户对该角色的经历下）。
- **物理存储：SQLite（经 MemoryPort）**，**禁止**把万级 episodic 塞进 `profile.save.json`。  
  - **存得下 ≠ 查得起、注得起**：全量可在库中增长；每通 context **不得**全量加载。
- **重做模型**，不照搬旧话机散落目录；也不照搬旧仓「截前 N 条、无检索」注入。
- CallCard.context 是**本轮剧本合同**，不是长期记忆。
- 通话中（含刚接通）Memory **只读**；写入只允许：
  1. **出口 Effect**：`patch_memory`
  2. 挂机 **`MemoryCommit`**
- **凡写入必须带时间戳**（见 §2.2）。
- Content（故事/角色 Def）继续 JSON；Memory 与 Content **分库分责**（见 [05](../技术设计文档/05-技术选型与工程边界.md)、[18](../技术设计文档/18-部署拓扑与BS架构.md)）。

## 1.1 跨角色记忆（已拍板）

挂机 / 卡消费完（Exit → Effect）后：

- **允许**对任意 `agentId` 执行 `patch_memory` / `update_npc_knowledge`（含非本通通话方）。
- **禁止**在 onConnect 或通话进行中改任何角色的 Memory / knowledge / facts（媒介类 `play_system_prompt` 除外）。

若下一通才需要对方「带着话」上场、又不想立刻改长期记忆：用 `attach_call_card` + 目标卡 brief（内容在卡上，不写 Memory）。

## 2. 分层（v1 定名）

```ts
/** 所有记忆条目的时间底线；实现校验拒绝缺时间戳的写入 */
interface MemoryTimestamps {
  /** 首次写入（ISO 8601，建议含 offset） */
  createdAt: string
  /** 内容被修改时更新；仅追加类可与 createdAt 相同 */
  updatedAt: string
}

interface CharacterMemory {
  /**
   * 逻辑聚合视图（调试/API 组装用）；物理行在 SQLite，不进 profile.save.json 大数组
   */
  profileNotes?: ProfileNotesMemory
  /**
   * 情节 / 会话摘要 + 生活边角碎片（可跨通累积）
   * 覆盖旧话机 last_session 摘要与 shared_memories 类「可再聊」素材
   */
  episodic: EpisodicEntry[]
  /** 结构化信念 / 稳定兴趣；尽量对齐 factId */
  semantic: SemanticBelief[]
  /** 对用户或其它角色的关系轴 */
  relational: RelationalState
  /** 当前情绪与待跟进，须带 TTL */
  affect: AffectState
  /** 约定 / 未完成事务（主观侧；研究任务真源在 Profile.research） */
  commitments: Commitment[]
}

interface ProfileNotesMemory extends MemoryTimestamps {
  text: string
}

interface EpisodicEntry extends MemoryTimestamps {
  id: string
  summary: string
  /**
   * 事件或本通结束时刻（通末摘要通常 = session endedAt）
   * 与 createdAt 可相同；保留 at 便于「事件何时发生」与「何时入库」分离
   */
  at: string
  callId?: string
  /**
   * call_summary：本通总结（聊了啥）
   * vignette：零散生活碎片（如「奶奶做了小蛋糕」），供以后找话题
   * beat：剧情节拍相关（少用）
   */
  kind?: "call_summary" | "vignette" | "beat"
  importance?: number
}

interface SemanticBelief extends MemoryTimestamps {
  id: string
  text: string
  factId?: string
  confidence?: number
}

interface RelationalState {
  towardUser?: RelationalEdge
  towardAgents?: Record<string, RelationalEdge>
}

interface RelationalEdge extends MemoryTimestamps {
  familiarity?: number
  trust?: number
  notes?: string
}

interface AffectState extends MemoryTimestamps {
  mood?: string
  followUp?: string
  expiresAt?: string
}

interface Commitment extends MemoryTimestamps {
  id: string
  text: string
  status: "open" | "done" | "cancelled"
  dueAt?: string
}
```

相对旧模型：层少、可过期、可对齐 factId、写入路径统一；**生活边角与通末总结统一进 episodic**，不再另开 shared_memories 仓。

### 2.1 episodic 产品口径（定稿）

`episodic` **同时**承担：

| 用途 | 说明 | 例 |
|------|------|----|
| 通末总结 | 一趟对话下来「聊了啥」（可称情节/会话摘要） | 「聊了家里近况，用户很开心」 |
| 生活边角 | 无任务、无履约，但可再提起的碎片；跨通累积 | 「奶奶给用户做了小蛋糕」 |
| 剧情节拍 | 可选；强剧情仍以卡 beats / Effect 为准 | `kind: "beat"` |

- **不要**把生活边角塞进 `commitments` / `Profile.research`（那些是履约真源）。
- 稳定偏好（「爱吃甜食」）→ 另写 `semantic`；一次性小事 → `episodic`（`vignette` 或通末 summary 内提及均可）。
- MemoryCommit：v1 **至少**一条本通 `call_summary`；LLM 可再抽 0–N 条 `vignette`。

对标旧话机：`last_session` ≈ 通末 `call_summary`；`shared_memories` 生活条 ≈ `vignette`；不迁旧文件结构。

### 2.2 时间戳铁律（定稿）

```text
凡写入 Memory 的条目 / 块 → 必须有 createdAt + updatedAt（ISO）。
缺时间戳 → 校验/运行时拒绝写入（或实现层自动补 now 并打 warning，v1 推荐拒绝以逼实现自觉）。
```

| 层 | 时间字段 |
|----|----------|
| 全部条目 | `createdAt`、`updatedAt` 必填 |
| episodic | 另有 `at`（事件/通话结束时刻） |
| affect | 另有可选 `expiresAt`（TTL） |
| commitments | 另有可选 `dueAt` |

**用途：**

1. Composer 投影时可标注「三天前聊过…」，LLM 知道新旧  
2. 后续特性：久未提起再聊、纪念日、过期淡化、调试按时间轴回放等  

`projectForCall` 应把相对时间（或绝对 ISO）一并注入 soft 段，而不是只丢无时间的纯文本。

## 2.3 注入分层（存全量 ≠ 注全量）

```text
热层（每通默认进 Composer soft）
  · 最近 N 通 call_summary（建议 N=3～5）
  · 最近 M 条 vignette（建议 M=5～10）
  · 当前 affect（未过期）+ relational 摘要 + profileNotes
  · 最近滚动「温层摘要」1 段（若有）
温层（库内预计算，不扫全表现算）
  · 按月/季的 rollup 摘要行（见 §2.4）
冷层（默认不进 prompt）
  · 更早 episodic 明细 → 仅经 recall / search 工具按需取
```

**硬预算（v1 建议，可配置）：**

| 项 | 建议默认 |
|----|----------|
| 热层总字符 | ≤ 1500～2500 |
| 单条 summary/vignette 注入 | ≤ 200 字（超出截断加省略） |
| 温层摘要段 | ≤ 400 字 |
| 禁止 | Connect/beginCall 时 `SELECT *` 全表或把整库 JSON.parse 进内存拼 prompt |

旧仓问题对照：只截「上一通 + 前 5 条共同经历、无时间、无检索」→ 本设计用热层+时间戳+只读召回替代。

## 2.4 老记忆总汇（rollup）

| 项 | 定稿（初步） |
|----|----------------|
| **汇什么** | 过热层窗口的 episodic → 生成/更新「月度或季度」摘要行（仍带时间范围戳） |
| **何时汇** | ① MemoryCommit 成功后**增量**尝试（本通若跨月/季边界则触发）；② 可选后台任务补算；**禁止**在 beginCall 热路径全量重算十年 |
| **存在哪** | SQLite 表 `memory_rollups`（见技术设计 [20](../技术设计文档/20-记忆存储与投影.md)） |
| **注什么** | 默认只注入**最近 1～2 段**温层摘要；更早靠 `search_memory` |

## 3. 与 WorldFact / knowledge

- 客观、可共享、需多角色可见的 → `Profile.world.facts` + `knowledge[agentId]`（effect：`set_world_fact` / `update_npc_knowledge`）。
- 主观感受、语气、私密情节、生活边角 → Memory 各层（effect：`patch_memory` / MemoryCommit）。
- Semantic 可引用 `factId`，避免两套真相；**升格为世界事实必须走** `set_world_fact`。

## 4. MemoryPort（引擎端口）

逻辑类型见 §2；**读写一律经本端口**，底层 SQLite（可换 Postgres 等，不改 Composer/Exit）。

```ts
interface MemoryPort {
  /** 只查热+近温；禁止全量加载 */
  projectForCall(input: {
    userId: string
    agentId: string
    card: CallCardDefinition
    nowIso?: string
  }): Promise<MemoryProjection>

  /** 通话中只读召回（供 search_memory / get_memory_by_id） */
  search(input: MemorySearchQuery): Promise<MemorySearchHit[]>
  getById(input: {
    userId: string
    agentId: string
    entryId: string
  }): Promise<MemorySearchHit | null>

  applyPatch(input: {
    userId: string
    agentId: string
    layer: string
    op: string
    payload: unknown
  }): Promise<void>

  commitAfterCall(input: MemoryCommitInput): Promise<MemoryCommitResult>

  /** 增量 rollup；可在 Commit 后调用 */
  rollupIfNeeded?(input: {
    userId: string
    agentId: string
    endedAt: string
  }): Promise<void>
}

interface MemorySearchQuery {
  userId: string
  agentId: string
  /** 关键词与时间窗至少提供一个 */
  textQuery?: string
  fromIso?: string
  toIso?: string
  kinds?: Array<"call_summary" | "vignette" | "beat" | "semantic" | "rollup">
  maxResults: number   // 默认 5；硬上限 ≤ 10
}

interface MemorySearchHit {
  id: string
  layer: string
  kind?: string
  text: string          // search 列表 ≤200 字；getById ≤500 字
  at: string
  createdAt: string
}

interface MemoryProjection {
  softText: string
  includedEntryIds: string[]
  rollupIds?: string[]
  debug?: { hotCount: number; chars: number }
}

interface MemoryCommitInput {
  userId: string
  agentId: string
  sessionId: string
  transcript: unknown
  outcome?: Outcome
  endedAt: string
}

interface MemoryCommitResult {
  ok: boolean
  writtenLayers: Array<
    "episodic" | "semantic" | "affect" | "commitments" | "relational" | "profileNotes"
  >
  writtenEpisodicIds?: string[]
  error?: string
}
```

**冷层召回定稿（摘要；算法细节见 [20 §5](../技术设计文档/20-记忆存储与投影.md)）：**

| 项 | 规则 |
|----|------|
| 存储检索 | SQLite **FTS5** 字面检索（可 LIKE 降级）；**v1 不做向量** |
| 隔离 | 仅本 `userId` × 本 `agentId`；Host 注入，禁跨角色 |
| 条件 | `textQuery` 与时间窗至少一组；`maxResults` 默认 5、硬 ≤10 |
| 回传 | search Hit 摘要 ≤200 字；getById 正文 ≤500 字 |
| 与投影 | 热层未带的旧记忆靠工具查；**禁止**为此把冷层灌进 beginCall prompt |

### 4.1 MemoryCommit 定稿

| 项 | 规则 |
|----|------|
| 命名 | 对外 **MemoryCommit**；端口方法 `commitAfterCall` |
| Free | 有效聊天后**默认开启**；可配置关闭 |
| Story | **默认关闭**；可选 `memoryCommit: "after_exit"` |
| 写层 | 至少一条 `call_summary`；可选 vignette / semantic；**写 SQLite**；必带时间戳 |
| 之后 | 可选 `rollupIfNeeded`（增量） |
| Free 顺序 | MemoryCommit →（有 candidate）ExitSelector → Effect |
| Story 顺序 | ExitSelector → Effect →（可选）MemoryCommit |
| 失败 | jsonl + 黄条；不回滚已执行 Effect |

| 实现 | 用途 |
|------|------|
| SqliteMemoryPort + server LLM | Studio / 壳默认 |
| Noop / Fixture | 调试与单测 |

## 5. 与 Composer 优先级

```text
hard: CallCard …
scene / time …
soft: MemoryProjection.softText（仅热+近温）+ Lore …
```

- Story 不得被记忆冲禁区。  
- Free 依赖 Memory + persona；**缺记忆时靠工具召回，不靠把冷层灌进 prompt**。

## 6. Studio

- 角色台 / 调试器：分页按时间查 SQLite（不要一次拉全表）
- Commit / Effect 后显示 diff
- 可模拟 `search_memory` / `get_memory_by_id` 查询

## 7. 非目标

- 接通时 / 通话中 **写** Memory
- 模型绕过 MemoryPort 直读库文件 / 直写
- 跨用户共享某角色记忆
- 无时间戳入库
- 另建 shared_memories 仓
- 用巨型 JSON Profile 存全量 episodic
- beginCall 时全量扫描或全量摘要重算
- `search_memory` 无 limit / 无时间窗关键词的裸全表扫并灌模型
