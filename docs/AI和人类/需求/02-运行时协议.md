# 02. 运行时协议

本文描述引擎运行时主链与算法约定。实现落在纯 TypeScript `packages/rpg-engine`，可被 Studio 与未来电话壳共用。

相关：[01](./01-CallCard数据模型.md) · [10](./10-用户与存档.md) · [19](../技术设计文档/19-引擎宿主与会话模型.md) · [50](./50-对话与模型适配.md) · [04](./04-调试器界面需求.md) · [语音留言改造需求](../里程碑/v2.0/语音留言改造需求.md)

## 1. 主链

```text
选定 User → 加载 PlayerProfile
CallIntent
  → CallCardResolver
  → 绑定 CallCardInstance；beginCall → CallSession（frozenCard + ComposeScene）
  → ContextComposer（只读；promptScenes + 本地时间）
  → DialogueAdapter.start(DialogueSessionSpec)
  → … DialogueEvent …（工具只登记 RuntimeExitCandidate）
  → session.ended + phoneFlags
  → Story：OutcomeEvaluator → ExitSelector → EffectExecutor（+ Sink）→ 可选 MemoryCommit
  → Free：FreeCallPostPipeline（MemoryCommit → 有候选则 ExitSelector → Effect）
  → （若 GenStack 非空）VoicemailMaterializePipeline（plan 终态之后；外置生成端口）
  → save Profile；WET / logs
```

铁律：

- 经历态只读写 **当前 PlayerProfile**；会话态在 **CallSession**（见 [19](../技术设计文档/19-引擎宿主与会话模型.md)）。
- **通话中不执行**推进类 effect；禁止 onConnect 写记忆。
- **active ≠ completed**；`call.ended ≠ story.card.completed`。
- Story 挂机后 Effect 可跨角色 `patch_memory`；Free 走总管线而非强制 exits。
- **`cardKind=voicemail` 永不进 `Board.pending`**；进信箱靠 attach/schedule 分流 + 物化（见 §8.4–§8.5）。
- 对话通道见 [50](./50-对话与模型适配.md)。

## 2. CallIntent（逻辑输入）

引擎不依赖话机 GPIO；只认逻辑意图：

```ts
type CallIntent =
  | { kind: "user_dial"; agentId: string }
  | { kind: "agent_outbound"; agentId: string; instanceId?: string }
  | { kind: "redial"; agentId: string }           // 来自重播槽
  | { kind: "simulate_start"; cardId: string }    // 调试器：从某卡强制开始
  | { kind: "mailbox_open"; agentId: string; voicemailId: string; cardId: string }
  // 信箱打开：听已物化留言；Resolver / beginCall 绑定该 voicemail 卡（entryMode=mailbox_open）
```

**听留言路径（钉死）：** 用户/调试打开信箱条目 → `mailbox_open` → `beginCall`（`frozenCard` = 该 `cardKind=voicemail` 卡，`interactionMode=playback_only`）→ 播槽内 text/audioRef → Outcome（`voicemail_listened` / `playback_completed` / `playback_skipped` 等）→ ExitSelector → 该卡 `exits`。**不推荐**壳自播完只打 flag、Host 另写无 Session 的 exits 捷径。
## 3. CallCardResolver

### 3.1 输入

- `intent: CallIntent`
- `board: CallCardBoard`
- `package: StoryPackage`
- `redialSlot?: { agentId: string; cardId?: string }`

### 3.2 规则（v1）

在 **当前 Profile** 上：

1. `simulate_start`：直接激活指定 `cardId` 的新 Instance（或已有 pending），忽略日常优先级。
2. `redial`：若槽指向 `cardId`，优先激活该 Definition 在目标角色上的对应 pending；若无 pending，可按包配置挂卡再激活（校验器应警告「重播未挂卡」）。
3. `mailbox_open`：按信箱槽 `voicemailId` + `cardId` 绑定该 **`cardKind=voicemail`** Definition；**不**从 `Board.pending` 挑选；Session 强制 `playback_only` + `entryMode=mailbox_open`。播槽内已物化内容。
4. `user_dial` / `agent_outbound`：
   - **有效可拨**：`effectiveDialable = !def.isNarrativeOnly && ((def.dialable === true) || (runtime?.unlocked === true))`。
   - 若意图为 `user_dial` 且 `!effectiveDialable` → `CHARACTER_NOT_DIALABLE`；`agent_outbound` / `simulate_start` 仍允许（**narrative-only 除外**：外呼亦拒绝，除非调试 force）。
   - **禁止**把 `cardKind=voicemail` 解析为待接通剧情卡（voicemail 永不在 pending）。
   - 在目标 `pending` 中筛选 `entryMode` 匹配的 Instance（拨号：`inbound_user_dial|either`（及 `inbound` 别名）；外呼：`outbound_auto|either`（及 `outbound` / `agent_outbound` 别名））。**`redialSlot` 仅作优先，不过 mode 则跳过**（见 `pickPendingForIntent`）。
   - 多张：比 `priority` 降序，同 priority 取 **createdAt 最新**。
   - 无匹配 pending：使用该角色 **FreeCallCard**（Session.`packageId = "__free__"`）。
5. 若该角色已有 `active` 且未结束：冲突（占线）。
6. ActiveStoryLock：按 [19 §8.4](../技术设计文档/19-引擎宿主与会话模型.md) 决策表（soft/hard × blockedPolicy）。

### 3.3 输出

```ts
interface ResolveResult {
  instance: CallCardInstance
  definition: CallCardDefinition
  source: "story_pending" | "free" | "redial" | "simulate" | "mailbox"
}
```

## 4. 会话期行为（逻辑）

| interactionMode | 引擎关注点 |
|-----------------|------------|
| `realtime_dialogue` | Composer + toolPolicy；经 DialogueAdapter 会话；等待结束 + Outcome |
| `playback_only` | 等待 `playback_completed` / `playback_skipped` / `hangup_early` / `missed` |
| `hybrid` | 先播放阶段，再进入对话阶段 |

通道（Realtime/文本/Manual）由 Adapter 决定，见 [50](./50-对话与模型适配.md)。

## 5. Outcome 构造

见 [01 §9](./01-CallCard数据模型.md)。v1 由调试器 / ManualEvaluator 填写；须符合同一 Outcome schema。

建议辅助计算：`missedRequiredBeats = requiredBeats - completedBeats`。
## 6. ExitSelector

### 6.1 算法

```text
candidates = exits.filter(exit => evaluate(exit.condition, outcome, world, variables))
if candidates.length == 0:
  → 返回 { ok: false, error: "no_exit_matched" }  // 校验器应尽量避免此态；调试器标红
else:
  → 按 priority 降序；同 priority 保持定义顺序
  → 取第一名
```

### 6.2 条件求值

对 [01](./01-CallCard数据模型.md) 中的 `ExitCondition` 递归求值。未知 op → 校验失败 / 运行时报错。

### 6.3 出口覆盖（校验边界）

- **story 卡**：完全无 `exits` → 校验 **warning** `EXIT_EMPTY_STORY`；有任意一条出口即不再因缺 `failure`/`recovery` 类别告警。
- **`exitKind`**：可选文档/标签字段；**不参与** ExitSelector（选出口只看 condition + priority）。Studio 属性面板默认隐藏该字段。
- **voicemail**：允许空 exits。

（创作上仍可自行为失败/恢复路径打标签，但非门禁。）

## 7. 逻辑 WET 事件

事件顺序与 Effect plan 失败策略详见 [16-错误处理与幂等事务](./16-错误处理与幂等事务.md)。摘要：

```text
call.ended（壳）
  → story.card.evaluated
  → story.card.exit_selected
  → story.effect.executed | skipped | failed
  → story.card.completed    // effect plan 终态之后
```

| 事件 | 何时 |
|------|------|
| `story.card.mounted` | attach 成功 |
| `story.card.unmounted` | unmount 成功 |
| `story.card.activated` | pending → active |
| `exit_candidate.registered` | 通话中登记动态出口 |
| `story.card.evaluated` | Outcome 已生成 |
| `story.card.exit_selected` | ExitSelector 已选定 |
| `story.effect.executed` | 单条 effect 成功 |
| `story.effect.skipped` | 幂等跳过 |
| `story.effect.failed` | 执行失败 |
| `story.card.completed` | **effect plan 终态之后**（含 completed_with_errors / aborted） |
| `story.card.exited` | 可选摘要事件（exitId + plan.status） |
| `story.ended` | `end_story` |

**禁止**把 `call.ended` 等同于 `story.card.completed`。

事件只追加，不改写历史。Board 由 Executor 更新时**必须同时**写事件。

动态出口见 [15](./15-动态出口与通话中工具事件.md)。

## 8. EffectExecutor

### 8.1 接口

```ts
interface EffectExecutor {
  execute(effect: Effect, ctx: ExecContext): Promise<EffectResult>
  executePlan?(effects: Effect[], ctx: ExecContext): Promise<EffectPlanResult>
}

interface EffectResult {
  ok: boolean
  skipped?: boolean
  error?: string
  summary?: Record<string, unknown>
}
```

`EffectPlanResult` / 失败策略见 [16](./16-错误处理与幂等事务.md)。v1：逐条执行，失败默认继续；`critical` 中止。

### 8.2 幂等

见 [01 §4.1](./01-CallCard数据模型.md)。ledger 存在模拟状态内；重复执行同一出口的同一 effect.id → `skipped: true`。

### 8.3 v1 Mock 行为

| effect | mock |
|--------|------|
| `attach_call_card` | **分流**：目标非 voicemail → Profile.Board 增加 pending；目标 voicemail → push VoicemailGenStack（**不写 Board**） |
| `unmount_call_card` | 移除/cancelled |
| `schedule_call_card` | **分流**：目标非 voicemail → 写入 once intent，到点 `agent_outbound`；目标 voicemail → once intent，到点入栈/物化进信箱（**非**外呼） |
| `set_redial_slot` | 更新模拟 `redialSlot`（可挂 Profile 或会话态） |
| ~~`create_voicemail`~~ | **废弃**：校验拒绝；运行时不执行新写。旧 stub 读路径可过渡，不得作为新写真源 |
| `keep_card_pending` | 当前 Instance 回到 pending（或保持 pending） |
| `set_world_fact` | 更新 Profile.world.facts |
| `update_npc_knowledge` | 更新 Profile.world.knowledge（可跨角色） |
| `patch_memory` | 更新 Profile.characters[agentId].memory（可跨角色；仅出口阶段） |
| `set_character_unlocked` | 更新 Profile.characters[agentId].unlocked |
| `update_user_profile` | 合并写入 Profile.user（nickname / fullName 等） |
| `schedule_recurring_call` | 写入 Profile.schedule.intents（`kind: "recurring"`） |
| `create_research_commitment` | 写入 Profile.research.commitments（见 [10](./10-用户与存档.md)） |
| `update_variable` | 更新 Profile.stories[].variables |
| `play_system_prompt` | 记一条日志事件 |
| `end_story` | 标记 StorySave completed |

真实话机替换 Executor 实现即可，**不改** ExitSelector / Resolver。

### 8.4 attach / schedule 对 voicemail 的分流

执行 `attach_call_card` / `schedule_call_card` 时须 **lookupCard** 解析目标 `cardKind`（见 [01 §4 分流表](./01-CallCard数据模型.md)）：

| 目标 kind | attach | schedule（到点） |
|-----------|--------|------------------|
| 非 `voicemail` | 写 `Board.pending` | `agent_outbound` 响铃语义（现状） |
| `voicemail` | push **VoicemailGenStack**；**禁止**写 Board | 入栈/物化进信箱；**禁止** `agent_outbound` |

**禁止：**

- 把 voicemail Instance 写入 `callCards.board.byAgent.*.pending`
- 在 `effectExecutor` 的 attach case 内直接调 LLM
- 挂机后隐式扫描 transcript 拼留言

### 8.5 VoicemailMaterializePipeline（plan 终态之后）

**时机：** Effect plan **终态之后**（含 `completed_with_errors` / `aborted`）。默认 **不丢弃**已入栈条目；已入栈仍应尝试物化。

**职责（Host 显式阶段，非 Executor 内联）：**

1. 出栈；加载卡定义；确认 `cardKind=voicemail`
2. 拼装生成输入（结构化；可含 prompt 字符串 + context 零件）
3. 调用注入端口 `generateVoicemail(input)`（引擎外 LLM/TTS；Studio Manual 可注入假实现）
4. 写 `Profile.telephony.voicemails[]`（text 与/或 audioRef）
5. 更新未读真源（由槽推导或显式布尔；**单一推导**）；调用 `onVoicemailUnreadChanged(hasUnread)`

**端口（与 EffectSink 并列；生成与 LED 分端口）：**

```ts
generateVoicemail?(input: {
  card: CallCardDefinition
  agentId: string
  packageId: string
  instanceId: string
  assembledPrompt?: string
}): Promise<{ text?: string; audioRef?: string }>

onVoicemailUnreadChanged?(hasUnread: boolean): void
```

失败：该条可标 `generate_failed` / 保持 pending；**不回滚**其它已成功 Effect。未注入生成端口时须有明确失败/跳过策略且不炸 Host。

未读真源在 **Profile.telephony**；壳仅订阅回调 / 读 Profile，**禁止**壳私自改 Board/Memory/未读真源。完整产品口径见 [语音留言改造需求 §3.4–§3.6](../里程碑/v2.0/语音留言改造需求.md)。

## 9. 调度回拨与时间

引擎维护逻辑时钟（调试器可控）：

```ts
interface SimulationClock {
  nowMs: number
  advance(ms: number): void
}
```

`schedule_call_card`：v1 **确定触发点** = `clockMs + minMs`（不随机）→ `ScheduledIntent.kind: "once"`。

- 目标 **非** voicemail：越过后生成 `agent_outbound`（外呼响铃）。
- 目标 **是** voicemail：越过后入 GenStack / 物化进信箱（**不是**外呼）；见 §8.4。

`redialSlot` 存 **Profile**（见 [19](../技术设计文档/19-引擎宿主与会话模型.md)、[10](./10-用户与存档.md)）。

## 10. 引擎对外 API

以实现级 Host 为准，见 [19 §1.2](../技术设计文档/19-引擎宿主与会话模型.md)。需求摘要：`resolve` / `beginCall` / `endCall` / `advanceClock` / `saveProfile` / `getSnapshot` / `getRecentLogs`。语音留言相关可注入：`generateVoicemail` / `onVoicemailUnreadChanged`（见 §8.5）。**禁止**依赖 React；**禁止**引擎持 LLM Key。

## 11. 与编辑器数据的关系

- Content：`characters/` + `storis-packages/*/story.conf.json` + `cards/*.s-card.json`；layout 旁车。
- React Flow = 投影；运行读写 **PlayerProfile + CallSession**，不是画布。
- 媒介 effect：Executor 写逻辑 + **EffectSink** 执行媒介（见 [19 §6](../技术设计文档/19-引擎宿主与会话模型.md)）。
- 留言正文物化：Host Materialize 写 Profile 槽；LED/提示音走壳端口（与生成分端口）。
